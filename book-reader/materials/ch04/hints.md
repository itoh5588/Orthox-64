# 第4章 段階的ヒント — メモリ管理を頭の中で迷子にならない

本章は写経して動かす章ではなく **「実コードを読み解く章」** です。手が止まりがちな箇所だけ Lv1 → Lv2 → Lv3 の順に開示してください。**まず本書本文を 4.1 〜 4.9 まで通読してから**、詰まった所だけここを参照することを強く推奨します。

ヒントは「どこを読めば自分で答えに辿り着けるか」を示すもので、答えそのものは出していません。各ヒントは具体的な `grep` コマンドや実コードの行番号目安を含みます。

## クラスタA — 物理アドレスと仮想アドレスの往復 (本書 4.2, 4.4)

本章で読者が最初につまづくのが「**いま自分が触っているのは物理アドレスなのか、仮想アドレスなのか**」の混乱です。ページテーブル自身もメモリ上のデータであり、それを編集するには仮想アドレスとしてポインタを持たないといけない、という入れ子構造が頭をぐらつかせます。

### Lv1 — 最初に確認すること
- `include/vmm.h` で `PHYS_TO_VIRT` と `VIRT_TO_PHYS` マクロの定義を確認 (`grep -n "PHYS_TO_VIRT\|VIRT_TO_PHYS" ../include/vmm.h`)。両方とも単純な加算/減算のはずです。
- 加算/減算しているオフセット (HHDM offset) はどこで設定されているか？ `grep -n "hhdm" ../kernel/` で追えます。

### Lv2 — それでも進まない場合
- HHDM offset は Limine ブートローダから渡される定数で、**カーネル起動後は不変**。だから「物理アドレス + 固定オフセット = 仮想アドレス」という1対1の写像で済みます。
- ページテーブル本体を編集するときの典型パターン: `void* pml4_phys = pmm_alloc(1);` → `uint64_t* pml4_virt = (uint64_t*)PHYS_TO_VIRT(pml4_phys);`。これを 4 階層ぶん繰り返しているだけ。
- `pmm_free` には**物理アドレス**を渡す必要がある (実コード `pmm_free` の引数を grep して確認)。「仮想ポインタを保持していたら、free 時に必ず `VIRT_TO_PHYS` で物理に戻す」が鉄則。

### Lv3 — 典型的な誤解
- 「HHDM は仮想 → 物理を行う特別な仕組み」と説明する解説書もありますが、**Orthox-64 では単なる加算オフセット**です。ハードウェアの何かを使っているわけではありません。
- 「CR3 には仮想アドレスを書く」は誤り。CR3 は **PML4 の物理アドレス** を保持します (本書 4.7 と `kernel/task.c` の `t->ctx.cr3` への代入で確認可能)。
- AI が「PHYS_TO_VIRT はカーネル内部の MMU 操作」と説明したら誤り。`include/vmm.h` のマクロ定義をその場で見せて訂正させてください。

## クラスタB — PMM の倉庫番モデル (本書 4.3)

PMM は**物理ページの倉庫番**であり、仮想アドレスを一切知らない、というのが本書のキーアイデアです。ここでつまづく読者は「`pmm_alloc(1)` が返す物理アドレスを、なぜそのままポインタとして使えないのか」を理解できていません。

### Lv1 — 最初に確認すること
- `include/pmm.h` で `pmm_alloc` と `pmm_free` の宣言を確認 (`grep -n "pmm_alloc\|pmm_free" ../include/pmm.h`)。返り値型と引数の単位は何か？
- `kernel/pmm.c` の `bitmap` と `ref_counts` の宣言箇所 (本書 4.3 のコード片) を grep。

### Lv2 — それでも進まない場合
- `pmm_alloc(n)` の引数は **「ページ数」**。バイト数ではありません。`pmm_alloc(1)` で 4 KiB、`pmm_alloc(4)` で 16 KiB。
- 返り値は `void*` ですが、**これは物理アドレスを `void*` の型に詰め直したもの** であり、デリファレンスしてはいけません。「物理アドレスを保持するためのバケツ」として `void*` を使っているだけ。
- `ref_counts` は COW のために必須。本書 4.3 末尾と 4.7 で再登場します。「`fork` した瞬間に親子で物理ページを共有 → どちらかが書き込み → ページフォルトで複製」の **「共有を後から検知する」** ための仕掛け。

### Lv3 — 典型的な誤解
- 「PMM が仮想アドレスを返してくれれば直接使えるのに」と思いがちですが、**PMM が仮想アドレスを知った瞬間にレイヤ違反**になります。VMM 側がページテーブルを介して各タスクに異なる仮想アドレスをマップするため、PMM 単体では「どの仮想アドレスにマップされるか」を決められません。
- AI が `pmm_alloc(4096)` のように「バイト数」で説明してきたら、`include/pmm.h` の宣言をその場で見せて誤りを指摘させてください。

## クラスタC — 4段ページテーブルの掘削 (本書 4.5, 4.6)

`vmm_map_page()` が PML4 → PDP → PD → PT を順に掘り進む、という構造は図 4.1 で説明されています。実コードを読むときの落とし穴は **「中間テーブルが動的に確保される」** ことです。

### Lv1 — 最初に確認すること
- `kernel/vmm.c` の `vmm_map_page` 本体 (line 144 付近) と、その中で呼ばれている `get_next_level` (line 99 付近) を読む。
- ビットフィールド分割マクロ (`PML4_IDX`, `PDP_IDX`, `PD_IDX`, `PT_IDX`) の定義を grep (`grep -n "PML4_IDX\|PDP_IDX\|PD_IDX\|PT_IDX" ../include/vmm.h`)。

### Lv2 — それでも進まない場合
- 各レベルのテーブルは「**64-bit エントリ 512 個 = 4096 バイト = 1 ページ**」。だから中間テーブルも `pmm_alloc(1)` 1 回で確保できる。
- `get_next_level` の `allocate=true` 引数は「該当レベルのテーブルが存在しなければ動的に作る」のスイッチ。これがあるおかげで、256 TB の仮想空間全部のテーブルを事前に作る必要がない (実際に使う部分だけテーブルが生える)。
- 仮想アドレス `vaddr` を 9 ビット × 4 + オフセット 12 ビット = 48 ビット で分割。x86_64 は本来 64 ビットだが、上位 16 ビットは符号拡張で「カノニカルアドレス」になる。

### Lv3 — 典型的な誤解
- AI が「PML4 のサイズは固定で、すべてのプロセスで共有される」と説明したら誤り。**PML4 はプロセスごとに新規 `pmm_alloc(1)` で確保される** (本書 4.7 と `kernel/task.c` の `task_create_on_cpu` 周辺で確認)。共有されるのは **PML4 の上位半分 (インデックス 256 以上、カーネル空間)** だけ。
- 「`vmm_map_page` は1回呼ぶと自動的に大量のページがマップされる」と思いがち。実際は **1 ページ (4 KiB) だけ** マップする。連続マッピングは呼び出し側の for ループ。

## クラスタD — PTE フラグの分類 (本書 4.6)

`PTE_PRESENT`, `PTE_WRITABLE`, `PTE_USER`, `PTE_NX` は x86_64 ハードウェアの仕様 (Intel SDM Vol 3A Chapter 4) で定義されています。一方 **`PTE_COW` は Orthox-64 が独自にソフト用ビット (bit 9) に定義したもの** です。これを混同するとAIが誤回答しがちです。

### Lv1 — 最初に確認すること
- `include/vmm.h` で `PTE_` 系定数の定義を全部 grep (`grep -n "PTE_" ../include/vmm.h`)。
- 各ビット位置を見て、bit 0, 1, 2, 63 がハード仕様、**bit 9 が `PTE_COW` (Available)** であることを確認。

### Lv2 — それでも進まない場合
- Intel SDM では PTE の bit 9-11 と bit 52-58 が **「Available for software use」** と明記されている。OS 設計者は自由にここを使ってよい。
- `PTE_COW` を立てると同時に、本書の説明通り `PTE_WRITABLE` を **クリア**する。`(*pte & ~PTE_WRITABLE) | PTE_COW`。これにより書き込み時に必ずページフォルトが発生し、ハンドラに制御が渡る。
- `PTE_NX` は EFER の NXE ビットを有効化していないと CPU が認識しない。`syscall_init_cpu` 周辺で EFER 設定を確認するのが面白い演習。

### Lv3 — 典型的な誤解
- AI が「CPU が PTE_COW ビットを見て自動的に複製する」と説明したら **完全な誤り**。CPU は PTE_COW を**全く認識しない**。ソフトビットは OS が読むためだけのもの。
- 「PTE_USER が 1 ならユーザーが自由に読み書きできる」と思いがちだが、PTE_WRITABLE と組み合わせないと書き込みはできない。両方 1 で初めてユーザー書き込み可。

## クラスタE — COW とページフォルトハンドラ (本書 4.7)

COW (Copy-on-Write) は本書のハイライトの1つです。「親子で物理ページを共有 → どちらかが書き込み → ページフォルト → 複製」のフローが頭の中で組み立てられるかが鍵。

### Lv1 — 最初に確認すること
- `kernel/vmm.c` の `vmm_copy_pml4` (line 209 付近) を読む。`fork` 時にこの関数が呼ばれ、**親子両方の PTE で `PTE_WRITABLE` をクリアし `PTE_COW` を立てる** ループが見えます。
- `vmm_page_fault_handler` (line 371 付近) を読む。**CR2 レジスタからフォルトアドレスを取得し、その PTE が `PTE_COW` 立ちなら複製処理**へ進む。

### Lv2 — それでも進まない場合
- ページフォルトハンドラの判断順序: (1) CR2 から仮想アドレス取得、(2) 該当 PTE を 4 段下って取得、(3) PTE_COW が立っているか確認、(4) 立っていれば `pmm_alloc(1)` で新ページを確保しコピー、(5) ref_count を見て「自分が最後の参照者ならコピー不要、PTE_WRITABLE を立て直すだけ」を判定。
- (5) の最適化が重要。`ref_count == 1` なら共有者は自分だけ → コピーせず単に書き込み許可を復活。AI がこの最適化を出さないコードを書いてきたら指摘可。
- 「`fork` 直後に親子で同じ物理ページを共有」している状態は、ページテーブルレベルでは **両方の PML4 配下の PTE が同じ物理アドレスを指している** ことに相当する。`vmm_copy_pml4` は中間テーブル (PDP/PD/PT) は新規に確保するが、最末端の物理ページは共有。

### Lv3 — 典型的な誤解
- 「ページフォルトハンドラは Linux 風に `mm/fault.c` にある」は誤り。Orthox-64 では `kernel/vmm.c:371` 付近。AI が架空のパスを出したら必ず grep で訂正させる。
- 「`vmm_copy_pml4` は新しい PML4 のポインタを返す」と AI が言ったら誤り。`include/vmm.h` の宣言を見ると **`uint64_t` (物理アドレス) を返す**。これは `task_create_on_cpu` 内で `t->ctx.cr3` に直接代入される。
- 「カーネル空間のコピーは `vmm_setup_kernel_mappings` 関数で行う」と AI が言ったら **その関数は存在しません**。`grep -n vmm_setup_kernel_mappings ../kernel/` で空であることを確認させる。実際はインラインの for ループ (本書 7.4 と `task_create_on_cpu` 内)。

## クラスタF — `sys_brk` と `sys_mmap` の共通基盤 (本書 4.8)

ユーザーランドから見ると `brk` と `mmap` は別物ですが、**カーネル内部ではどちらも最終的に `pmm_alloc + vmm_map_page` に降りていきます**。これが本章の締めくくり。

### Lv1 — 最初に確認すること
- `kernel/sys_vm.c` で `sys_brk` (line 168 付近) と `sys_mmap` (宣言 line 165) を読む。
- それぞれの実装内で `pmm_alloc` と `vmm_map_page` がどう呼ばれているかを目視。

### Lv2 — それでも進まない場合
- `sys_brk` はプロセスの `heap_break` を押し広げ、押し広げた範囲を `vmm_map_page` で 1 ページずつマップ。`PTE_USER | PTE_WRITABLE | PTE_PRESENT` の組み合わせがユーザー書き込み可能ヒープページの最小契約。
- `sys_mmap` は anonymous mapping と file-backed mapping の両方を扱うが、Orthox-64 では**遅延ページインを実装せず、マップ時に物理ページを確保してファイルを一括コピー**する素朴設計 (本書 4.8 にも明記)。
- ゼロクリア (`kernel_memset(...)`) は**セキュリティ要件**。他プロセスが解放した物理ページの内容が残っていると情報漏洩する。本書のコード片を見落とすと AI が「ゼロクリアは不要」と言ってくる可能性。

### Lv3 — 典型的な誤解
- 「`sys_mmap` で確保したページはアクセス時に初めて物理メモリが割り当てられる (デマンドページング)」は **Orthox-64 では誤り**。マップ時即確保。Linux と混同しないこと。
- 「`brk` は古い syscall でもう使われない」は誤り。musl libc / BusyBox / Python の起動初期は **`brk` で初期ヒープを取る** ことが多い。本書 4.9 で明示。

## AI への聞き方のコツ

本書 4.10「AI アシスタント」節の質問例をそのまま使うのが基本ですが、以下の追加ルールを守ると精度が上がります。

- **シンボル名を必ず添える**: 「`vmm_map_page`」「`PTE_COW`」「`vmm_page_fault_handler`」のように実コードのシンボル名を質問文に入れる。AI が概念だけで答えると Linux 知識と混同しがち。
- **「実コードを `grep` で確認してから答えて」と必ず添える**: AI は本書の概念コードをそのまま実コードと信じがち。グラウンディングを強制する。
- **架空の関数名を見抜く**: `vmm_setup_kernel_mappings` のような「ありそうだが存在しない」関数名を AI が出してきたら、必ず `grep -n` で実体を確認させる。
- **ハードビット vs ソフトビットの区別**: PTE フラグの説明で AI が混同していないかチェック。`PTE_COW` は CPU が認識しないソフトビット。
