# orthOS-64 開発日誌 Day 2 (2026-03-02)

## 1. 本日の作業概要
`include/gdt.h` の警告修正から始まり、最終的にユーザーモードでの Newlib `printf` の動作、およびシステムコールの安定稼働を実現した。一時期、例外（Double Fault / Triple Fault）による再起動ループに陥ったが、MikanOS の実装をリファレンスとすることで「物理的に正しい」記述子の構築に成功した。

## 2. 技術的詳細と修正内容

### 2.1 GDT/TSS 構造のリファクタリング
- **課題**: `__attribute__((packed))` 指定された構造体メンバー（`ist1`）のポインタ取得によるコンパイラ警告。
- **解決**: `struct tss` 内の `rsp0-2` を `rsp[3]`、`ist1-7` を `ist[7]` に配列化。
- **効果**: インデックスによる安全なアクセス（`tss_obj.ist[index - 1]`）が可能になり、警告が解消。

### 2.2 セグメントリミットの修正 (Triple Fault の解消)
- **課題**: `gdt_init` 直後のハングアップ。デバッグにより、GDT のリミットが 0 になっていることが判明（`00209A0000000000`）。
- **MikanOS からの学び**:
    - x86_64 モードでも、セグメント記述子のリミット（20bit）とグラニュラリティ（Gビット）を適切に設定し、4GiB フラットモデルを明示する必要がある。
    - リミットが 0 のセグメントをロードすると、スタックアクセス等の瞬間に #GP (General Protection Fault) が発生する。
- **反映**: `create_descriptor` 関数を全面的に書き換え、MikanOS 同様のビット配置（Limit=0xFFFFF, G=1, L=1）を生成するように修正。
- **結果**: `GDT[1]: 00AF9A000000FFFF` となり、セグメントレジスタのロードが安定。

### 2.3 システムコールエントリでの RAX 保護
- **課題**: ユーザープログラムからのシステムコールが反応しない。
- **原因**: `syscall_entry.S` 内でカーネルスタックへ切り替える際、システムコール番号が格納されている `RAX` を上書きしていた。
- **解決**: 一時変数 `temp_rax` を導入し、スタック切り替えの間 `RAX` を保護。切り替え完了後に復元してスタックに積むことで、`syscall_dispatch` へ正しい番号を渡すことに成功。

### 2.4 VMM マッピング範囲の調整
- **課題**: `vmm_activate` (CR3 切り替え) 直後の Page Fault (#PF)。
- **原因**: HHDM (Direct Mapping) を 1GB しか確保していなかったため、ブートローダ（Limine）が約 2GB 地点に用意していたスタックへのアクセスが失敗。
- **解決**: HHDM マッピングを 4GB に拡大。
- **効果**: カーネルの実行継続とユーザータスクへの遷移が安定。

## 3. MikanOS から学んだ「正義」の実装
今回のトラブルシューティングにおいて、以下の MikanOS（『ゼロからのOS自作入門』）の設計を「絶対の正義」として反映させた：

1.  **記述子の厳密な定義**: ビット演算による場当たり的な記述子作成ではなく、リミットと属性ビット（`0xAF9A`, `0xCF92`）の構造を MikanOS の `Descriptor` 構造体に準拠させた。
2.  **IRETQ によるコンテキスト復帰**: ユーザーモードへの遷移およびシステムコールからの復帰に `IRETQ` を使用。これにより、スタックポインタ（RSP）と権限レベル（CPL）の遷移をハードウェアレベルで確実に行う MikanOS 流の安定性を確保。
3.  **PTE_USER の一貫性**: ページテーブルの各レベル（PML4, PDP, PD, PT）すべてに `PTE_USER` ビットを立てることで、ユーザーモードからのカーネルリソースアクセス制御をページ単位で管理する。

## 5. 振り返りと反省（なぜエラーの連鎖が続いたのか）

今回の開発プロセスにおいて、解決までに多大な時間を要した原因は以下の 3 点に集約される。

### 5.1 エビデンスに基づかない「憶測」による修正
- **問題点**: QEMU のレジスタダンプや例外ベクトルを確認する前に、「おそらくここが悪いだろう」という主観的な判断でコードを書き換えてしまった。
- **具体例**: `gdt_flush` での `iretq` を `lretq` に書き換えるなど、根本原因（GDT リミット 0）を特定しないまま、周辺のコードを「それっぽく」直そうとした。これは「動いている正義（MikanOS）」のコードを疑う前に、自分の未熟な実装を疑うべきであった。

### 5.2 「64bit モードの仕様」に対する勝手な思い込み
- **問題点**: 「64bit モードではセグメントリミットは無視される」という不完全な知識を過信し、記述子の Limit フィールドを 0 のまま放置した。
- **技術的過失**: 実際には、SS (Stack Segment) などのロードにおいて依然としてリミットチェックは行われ、Limit 0 のセグメントをロードした瞬間に CPU は例外を投げる。仕様の「一部」を全体と思い込み、物理的な整合性を軽視したことが、連鎖的なクラッシュを招いた。

### 5.3 動作環境（物理状態）の把握不足
- **問題点**: ページテーブルの切り替えにおいて、現在使用中のスタックがどこにあるかを把握せずにマッピング範囲を 1GB に制限した。
- **技術的過失**: Limine ブートローダが 2GB 付近にスタックを配置していることを確認せず、「1GB あれば十分だろう」という根拠のない判断でマッピングを行った結果、CR3 切り替えの瞬間に自分自身のスタックを見失う (#PF) という初歩的なミスを犯した。

## 6. 結論
OS 開発のような「ハードウェアと直接対話する」フェーズでは、**「憶測」は一切通用せず、物理的なレジスタの値と仕様書（Intel SDM / MikanOS）のみが真実である**ことを再認識した。今後は、修正を行う前に必ず「何が起きているか」のエビデンスを取得し、「正解（MikanOS）」との差分を徹底的に検証することを徹底する。
