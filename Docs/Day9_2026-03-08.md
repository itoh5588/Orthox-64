# Day 9 作業ログ (2026-03-08)

## 本日の実装内容と達成事項

本日は、orthOS における「システムの堅牢化」と「メモリ効率の向上」という、商用OSに近いレベルの基盤改善を達成しました。

### 1. プリエンプティブ・マルチタスクの実現
これまでの協調的マルチタスクを卒業し、OSが主導権を握るプリエンプティブ・マルチタスクを実装しました。
- **LAPICタイマーの活用**: LAPICタイマーを 100Hz (10ms) の周期で割り込みが発生するように設定。
- **割り込みハンドラからのスケジューリング**: タイマー割り込み（Vector 32）が発生するたびに強制的に `schedule()` を呼び出し、タスクを切り替える仕組みを構築。
- **定数化**: `include/lapic.h` を新設し、`SCHED_TICK_MS` などの定数で切り替え時間を管理できるようにマジックナンバーを排除。
- **検証**: 無限ループするプログラム (`loop.elf`) を実行中も、シェルや他のタスクが動き続けることを実証。

### 2. Copy-On-Write (COW) の実装
`fork()` 時のメモリ消費を劇的に抑え、システムの安定性を高める COW を実装しました。
- **参照カウントの導入**: 物理メモリ管理 (`pmm.c`) を拡張し、各物理ページがいくつのプロセスから参照されているかを追跡する `ref_counts` テーブルを追加。
- **共有ページの設定**: `vmm_copy_pml4` において、物理ページをコピーする代わりに、親・子のページテーブル双方を「書き込み禁止 かつ COWフラグあり」に設定。
- **ページフォルト・ハンドラ**: 書き込み違反（Vector 14）をフックし、参照カウントが 2 以上の場合は新しいページを割り当ててデータをコピー、1 の場合は単に書き込み許可へ昇格させるロジックを実装。
- **検証**: `cowtest.elf` を用いて、書き込みが発生した瞬間にのみ物理コピーが行われることを確認。

### 3. ELFローダの抜本的改善
Day 8 で発生していた `.rodata`（文字列定数など）が正しく出力されない問題を解決しました。
- **問題の原因**: 従来のローダはセグメント単位で一括して物理ページを割り当てていたため、同じページに重なる複数のセグメント（`.text` と `.rodata` など）があると、後からロードされたセグメントが前のデータを上書きして消失させていた。
- **解決策**: `vmm_get_phys` を新設し、ページ単位でロードを行うように変更。既にマップされているページがあればその物理アドレスを再利用してデータを追記するように修正。
- **結果**: ページ境界に揃っていないセグメントも精密にロードされるようになり、文字列表示の化けが解消。

### 4. ビルド環境とコードの整理
- **`build/` ディレクトリの活用**: ソースディレクトリに `.o` ファイルが散らばらないよう、`Makefile` を修正してビルド成果物を `build/` に集約。
- **警告の修正**: コンパイル時に出ていた未使用関数の警告などを一掃。

### 5. ページ保護不具合の原因切り分けと修正（追記）
`rotest.elf` で「読み取り専用データへの書き込み」を検証したところ、当初は期待と異なる挙動が見えたため、ログと実装を突き合わせて原因を切り分けました。

- **原因1（本質）**: `vmm_copy_pml4` が `PML4[0]` をコピーしておらず、`fork -> execve` 経路でユーザー空間（`0x400000` 帯）のページングが破綻していた。
  - 修正: ユーザー空間コピー範囲を `for (i = 0; i < 256; i++)` に変更。
- **原因2（検証ノイズ）**: ビルドエラー時に古い `orthos.iso` を起動してしまい、最新コードの検証結果が反映されていなかった。
  - 修正: ビルド不整合（宣言漏れ・型不一致）を解消して `make all` 完走後に再検証。
- **堅牢化**: ページフォルトハンドラにユーザーフォルト時のプロセス異常終了処理を追加し、PML4E/PDPE/PDE の `present` チェックを明示化。

**最終検証結果（`bash test_ro.sh` + `serial_output.log`）**
- `Attempting to write to read-only data...` の直後に  
  `#PF(User): write-to-nonwritable ... Error: 0x7` を確認。
- 該当子プロセスは `TASK_ZOMBIE` となり、親の `wait4` が回収。
- シェルはプロンプトへ復帰し、システム全体は継続動作。

これにより、「書き込み禁止ページへの書き込みは #PF で保護され、当該プロセスが異常終了する」ことを実機シーケンスで確認できました。

### 6. 引き継ぎ対応（追加改修）: ページング安定化 + VRAMマッピング実証
前回の修正後、ワークツリー上で一部退行（`PML4[0]` 非コピーや ELF 権限処理）が混在していたため、引き継ぎとして再度整合性を取り、DOOM移植向けの基盤まで実機確認しました。

- **ページング退行の再修正**
  - `vmm_copy_pml4` のユーザー領域コピーを `i=0..255` に修正し、`PML4[0]` を確実に継承。
  - `#PF` ハンドラにユーザーフォルト時のプロセス異常終了（`TASK_ZOMBIE`）を実装。
  - PML4E/PDPE/PDE の `present` チェックを追加し、フォルト処理を堅牢化。
- **ELF ローダ権限の是正**
  - `elf_flags_to_vmm` で `PF_W` を反映し、全セグメント writable だった不具合を修正。
  - これにより `.rodata` への書き込みは正しく #PF となる。
- **VRAM システムコール系のビルド/実行整備**
  - `syscall.c` に `limine.h` 依存と `kernel_memset` を補完し、ビルドを通過。
  - `testvram.c` の `video_info` 重複定義を解消。

**最終実機検証（QEMU + serial）**
- `rotest.elf`:
  - `Attempting to write...` 直後に  
    `#PF(User): write-to-nonwritable ... Error: 0x7` を確認。
  - 子プロセスが異常終了し、シェルは `$` へ復帰。
- `testvram.elf`:
  - `Video Mode: 1280 x 800, BPP: 32, Pitch: 5120`
  - `VRAM mapped at 0x1000000000`
  - 画面塗りつぶし完了後に `$` へ復帰。

これにより、Day9 の未完了だった「メモリ保護の再安定化」と「DOOM向け直接描画基盤」の両方を、実機ログベースで確認できました。

## 今後の課題・次へのステップ
- **ELFローダのさらなる改善**: 現在は全セグメントを `writable` でマップしているため、ELF のフラグ (`PF_R`, `PF_W`, `PF_X`) を反映したマッピングを行う。
- **システムコールの拡充**: `make` の移植に向けて、`sigaction` や `ioctl`、不足しているファイル操作系のシステムコールを調査・実装する。
- **Demand Paging (デマンドページング)**: COW の技術を応用し、プログラム起動時にすべてをメモリに載せるのではなく、アクセスがあった時に初めてディスクから読み込む仕組みへ進化させる。

## 午後6時以降の進捗：DOOM 移植の基盤構築

DOOM 移植の第一歩として、高解像度グラフィックスへのアクセス環境を整えました。

### 1. グラフィックス制御システムコールの追加
- **`SYS_GET_VIDEO_INFO` (100)**: 画面の解像度、ピッチ、ピクセルフォーマットを取得。
- **`SYS_MAP_FRAMEBUFFER` (101)**: 物理 VRAM をユーザー空間の固定アドレス（`0x1000000000`）にマッピング。
- これにより、DOOM の描画エンジンが OS を介さず高速に画面を更新できる土台が整いました。

### 2. 直面している課題：厳格な保護による副次的な Page Fault
- メモリ保護を MikanOS 準拠の堅牢な設計（中間テーブルの分離など）に更新した結果、今まで動いていたプログラムが起動時に `#PF` で停止するようになりました。
- **原因**: ELFローダがセグメントをマッピングする際、既存の中間テーブル（PML4E, PDPE, PDE）の `PTE_USER` 属性を更新しきれておらず、末端のページが User 用であっても、そこに至る経路が Supervisor 専用のままになっていたためです。
- **対策**: `elf_load` において常に `vmm_map_page` を使用し、中間階層への権限波及を強制する修正を検討中。

### 3. テスト環境の改善
- QEMU の `sendkey` 入力においてアンダースコアが不安定なため、テストバイナリ名を `testvram.elf` のように記号を排除した形式に変更。

## 夜間追記2：DOOM 移植の実装とビルド統合

先ほどの「DOOM 移植への挑戦開始」に基づき、`doomgeneric` のポーティング作業を進めました。

### 1. doomgeneric の取得と設定
- `user/doomgeneric` に doomgeneric リポジトリをクローン。
- orthOS 向けのハードウェア依存スタブとして `doomgeneric_orthos.c` を新規作成し、以下の関数を実装（初回ドラフト）：
  - `DG_Init()`: `SYS_GET_VIDEO_INFO` および `SYS_MAP_FRAMEBUFFER` を用いたVRAM初期化
  - `DG_DrawFrame()`: VRAM へのピクセルコピー
  - `DG_SleepMs()`, `DG_GetTicksMs()`: タイマーシステムコールの呼び出し
  - `DG_GetKey()`: イベントキューシステムコールを用いたキー入力取得

### 2. ビルド環境の統合
- `doomgeneric` の `Makefile` を改修し、ホスト環境のコンパイラではなく、orthOS用の `x86_64-elf-clang` でクロスコンパイルするように設定。
- orthOS のユーザー空間用ライブラリ (`crt0.o`, `syscalls.o`, `libc.a`, `libgcc.a`) をリンクするように変更。
- DOOM は内部で一部浮動小数点演算を使用するため、リンクエラーを回避する目的で `dummy_math.c` を追加し、単純な整数演算にフォールバックする仮実装を提供。

### 3. ISOイメージへの組み込みと初回起動テスト
- ルートの `Makefile` および `iso/limine.conf` を更新し、ビルドされた `doom.elf` を ISO イメージの `/boot/doom.elf` として配置。
- QEMU 上で `doom.elf` を実行するスクリプトを作成し、テストを自動化。

### 4. 直面している課題：システムコール呼び出し時のページフォルト
- 初回起動テストにおいて、`DG_Init` が呼ばれた直後に `#PF: Unexpected page fault at 0x0000000000000064` （アドレス 0x64）が発生しクラッシュ。
- 原因: `doomgeneric_orthos.c` 内部に記述した自作のインラインアセンブリ (`syscall` 命令) の制約指定に誤りがあり、システムコール番号である `100` (`0x64`) が、レジスタへの値セットではなくメモリアドレスへのアクセスとして誤解釈されていると推測。
- **次のステップ**: 自作のインラインアセンブリを破棄し、すでに実績のある `user/syscalls.c` のシステムコール・ラッパーを直接呼び出すように `doomgeneric_orthos.c` を修正する。
