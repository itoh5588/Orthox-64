# Day 6 作業ログ (2026-03-05)

## 本日の目標
Day 5 で構築した基盤の上に、「ユーザーとの対話機能（シェル）」と「ファイルシステム（TAR）」を実装し、実用的なOS環境の第一歩を築く。

## 達成された進捗

1. **インタラクティブシェル (`sh.elf`) の実装**
   - ユーザー空間で動作する本格的なシェルを `user/sh.c` に実装。
   - `fork` と `execve` を組み合わせた外部プログラムの実行、および `waitpid` による同期を実現。
   - `cat` コマンド相当の機能を内蔵し、ファイル内容の表示を可能にした。

2. **Cランタイム (`crt0.S`) と Newlib の統合**
   - ユーザープログラムの共通エントリポイント `crt0.S` を導入。`main(argc, argv)` への引数渡しと終了処理を標準化。
   - `user/syscalls.c` を更新し、Newlib が要求するアンダースコア付きのシステムコールスタブ (`_read`, `_write`, `_sbrk` 等) を完備。

3. **シリアルポート (UART) 入力の完全対応**
   - 8259 PIC の IRQ4 (Serial COM1) を有効化。
   - UART の初期化手順を修正（DLABビットの適切な制御）し、ハードウェア割り込みによる入力を実現。
   - シリアル入力バッファをカーネルに実装し、`sys_read` を通じたブロッキング入力を可能にした。

4. **USTAR (TAR) ファイルシステムの導入**
   - `include/fs.h` に TAR ヘッダ構造を定義。
   - `kernel/fs.c` の `sys_open` を拡張し、Limine モジュール内の `.tar` ファイルを走査して内部のファイルを展開・オープンする機能を実装。
   - これにより、OS 内で「ファイル名」によるディレクトリライクなアクセスが可能になった。

5. **ABI 準拠とスタックアラインメントの修正**
   - ユーザープログラム（特に Newlib）が SSE 命令を使用する際の GPF (例外 0x0D) を解決。
   - タスク生成時および `execve` 時のスタックポインタ (RSP) を 16 バイト境界に厳密にアラインさせるよう修正。

6. **書き込み可能ファイルシステム (Ramfs) の実装** (追加作業)
   - メモリ上にファイルを保持する `Ramfs` を VFS に統合。
   - `O_CREAT` フラグによる新規ファイル作成、および `sys_write` による書き込みに対応。
   - シェルに `touch` コマンドを追加し、ユーザー空間からのファイル生成・保存・再読み込み（`cat`）を確認。

7. **環境変数 (`envp`) の伝達サポート** (追加作業)
   - `task_execve` を拡張し、`argv` だけでなく `envp` 配列もユーザースタックにコピーするよう修正。
   - これにより、GCC 等の実行に必要なパス情報や環境設定を渡す土台が完成。

8. **GCC コンパイル・パイプラインのシミュレーション成功** (最重要マイルストーン)
   - シェルに `ls` コマンドを追加し、TAR と Ramfs の両方のファイルを一覧表示可能にした。
   - 4つのユーザープログラム（`gcc.elf`, `cc1.elf`, `as.elf`, `ld.elf`）を実装。
   - 実際の `fork` / `execve` / `wait` / `VFS` をフル活用し、ソースコードから中間ファイルを経て実行ファイル（ダミー）を生成する一連の流れが orthOS 上で完結。
   - コンパイラのような複雑なプログラム連携が可能なレベルまで OS の安定性が向上したことを実証。

## 設計とアーキテクチャの考察

### プロセス管理構造の近代化
- 伝統的な UNIX の `struct proc`（常駐情報）と `u 構造体`（実行時情報）を統合した **`struct task`** を採用。
- **xv6 との比較**: MIT の教育用 OS である xv6 も、スワップアウト機能を排したシンプル化のために `u 構造体` を持たず、`struct proc` に情報を集約している。orthOS-64 もこの効率的な設計を継承しつつ、さらに 64ビット環境に最適化させている。
- 64ビットの広いアドレス空間と豊富なメモリを活かし、管理情報を一箇所に集約することで、コードの複雑性を抑えつつ、プロセスの親子関係や資源（FD等）の管理を効率化。

### Per-CPU データと高速なコンテキストスイッチ
- x86_64 の **`MSR_GS_BASE`** を活用し、`gs` レジスタを Per-CPU データへのポインタとして利用。
- システムコール突入時に `swapgs` 命令を使用することで、現在実行中のタスクのカーネルスタック情報を瞬時に取得し、ユーザー・カーネル間の遷移を安全かつ高速に行う仕組みを確立。

### パイプ (`pipe`) 実装への道筋
- 本日の VFS 拡張（TAR/Ramfs の共存）により、`struct task` 内のファイル記述子テーブルが「抽象化されたリソース」として機能し始めた。
- これにより、メモリバッファを共有する新しいファイルタイプ（`FT_PIPE`）を追加することで、容易に `pipe()` システムコールを実装できる設計的見通しが立った。
