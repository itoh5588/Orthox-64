# Day 5 作業ログ (2026-03-04)

## 本日の目標
Day 4 で実現した `fork()` の基盤を足がかりとし、プロセス管理・メモリ管理・VFS・標準入力といった「OSとしての基本機能群」を整備し、シェルを動かすための土台を完成させる。

## 達成された進捗 (正義のコード)

1. **`execve` システムコールの実装**
   - ユーザー空間の新しい ELF バイナリを読み込み、現在実行中のタスクのページテーブルとアドレス空間を置き換える `task_execve` を実装。
   - プロセスの PID を維持したまま、新しいプログラムに制御を移すことに成功。
   - **コマンドライン引数の引き回し**: `task_execve` 実行時に、`argv` の文字列データとポインタ配列を新しいユーザーのスタック空間 (`0x7FFFFFFFF000` 直下) へ正しくコピーし、16バイト境界のアラインメントを維持したまま、System V AMD64 ABI に従って `rdi` (`argc`) と `rsi` (`argv`) を `main` 関数へ渡す仕組みを実装した。

2. **`waitpid` (sys_wait4) の実装とプロセス終了待機**
   - `struct task` に親プロセスの PID (`ppid`) と終了ステータス (`exit_status`) を追加。
   - `sys_exit` 呼び出し時にタスクの状態を `TASK_ZOMBIE` とし、終了ステータスを保持。
   - 親プロセスが `waitpid` を呼んだ際、子プロセスが `TASK_ZOMBIE` になるまで `schedule()` でブロッキングループを行い、終了後にステータスを回収してリソースを解放(仮)する仕組みを実装。

3. **`sbrk` (動的メモリ割り当て) の実装**
   - これまでダミーだった `sys_brk` を拡張し、ユーザープログラムからヒープ領域の拡張が要求された際に、`pmm_alloc` で物理ページを割り当て、ゼロクリアしたのちに `vmm_map_page` でページテーブルへマッピングする処理を追加。
   - `user_test.c` で `sbrk(8192)` を呼び出し、割り当てられたメモリへの書き込みと読み出しがページフォールトなしに行えることを確認。

4. **仮想ファイルシステム (VFS) 基盤の導入**
   - `fs.h` と `fs.c` を新設し、`struct task` 内にファイルディスクリプタ (FD) テーブルを実装。
   - Limine プロトコルによってメモリ上にロードされたモジュール (ELF バイナリ等) を「読み取り専用のファイル」として扱えるようにした。
   - `open`, `read`, `close` システムコールを実装し、ユーザー空間からファイルパス ("exec_test.elf" など) を指定して FD を取得し、メモリ上のデータを `read` で読み出せることを確認。

5. **キーボード入力 (stdin) の実装**
   - `kernel/pic.c` を追加して 8259 PIC を初期化し、IRQ1 (キーボード) のみをマスク解除。
   - `kernel/idt.c` と `kernel/interrupt.S` に `isr33` (ベクタ33 = IRQ1) を追加し、キーボード割り込みハンドラへディスパッチ。
   - `kernel/keyboard.c` で PS/2 ポート (`0x60`) からスキャンコードを読み取り、ASCII 文字に変換してリングバッファに保存する仕組みを構築。
   - `sys_read` にて、FD が 0 (`stdin`) の場合はキーボードバッファから読み出しを行い、バッファが空の場合は `schedule()` でブロッキングする処理を実装。QEMUモニター経由のキーストロークがユーザープログラム (`user_test.c`) でエコーバックされることを確認。

## 遭遇した問題と解決策
- **ページフォールト (Error Code: 0x04) の頻発**
  - `user_test.c` に `waitpid` や引数テストを追加した直後、`RSP=0`, `RIP=0x00007FFFFFFFEFF8` でクラッシュする問題が発生した。
  - **原因**: `main.c` のタスク生成処理において、Limine モジュールの配列 `modules[0]` を盲目的に `user_test.elf` として扱っていたが、Makefile の記述変更により `exec_test.elf` が先頭に来る場合があった。
  - **解決策**: `main.c` の初期化ループにおいて、パス名 (`user_test.elf`) のサフィックスマッチングで確実に目的の ELF ファイルを検索・ロードするように修正。

## 次回 (Day 6) の指針と展望

ユーザーの最終目標である **「GCC によるセルフコンパイル」** を視野に入れたロードマップへシフトする。

1. **Newlib (標準Cライブラリ) の本格統合**
   - GCC などの巨大なプログラムを動かすには、フルスペックの libc が必須である。
   - 現在の自作の簡易 `syscalls.c` ではなく、Newlib をコンパイルしてリンクし、`_open`, `_read`, `_write`, `_sbrk`, `_fstat`, `_isatty`, `_lseek` などのシステムコールスタブをカーネルのシステムコール (`syscall_dispatch`) に正しくマッピングする。

2. **TAR ファイルシステムの展開と本格的な VFS**
   - GCC 本体やヘッダファイル群を OS 内に持ち込むため、Initrd (初期 RAM ディスク) として TAR アーカイブを読み込めるようにする。
   - パス解決（`/bin/gcc`, `/usr/include/stdio.h` など）やディレクトリ構造 (`stat`, `readdir`) をサポートする。

3. **シグナルとパイプ (必要に応じて)**
   - GCC は内部で `cc1` や `as`, `ld` を `fork` / `execve` し、標準入出力をパイプ等で繋ぐ場合がある。
   - シェルの高度化と併せて、パイプ (`pipe()`) や簡単なシグナル処理を検討する。

まずは明日の第一歩として、**シェルプログラムの実装** と **Newlib の本格稼働** に取り組み、ユーザー空間の開発環境をリッチにすることから始める。
