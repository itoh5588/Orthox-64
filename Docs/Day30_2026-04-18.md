# 日報: musl BusyBox smoke 完遂と fork/FD 破壊バグの修正 (Day 30)
**日付: 2026年4月18日**

## 1. 本日の目標
- Day 29 時点で残っていた `muslbusyboxsmoke` の最終クラッシュを解消し、テストを `PASS` させる。
- `ash` からの `fork` / `execve` / `wait4` / ファイル操作を含む一連の musl 実行経路を安定化する。

## 2. 実施内容

### 2.1 `fork` 後の FD 継承バグを特定・修正
- **根本原因の特定**:
    - `task_fork()` が `file_descriptor_t` を基本的に浅いコピーで継承していた。
    - `FT_DIR` / `FT_RETROFS` の `data` バッファは親子で共有されたままになっており、後続の `close()` で双方が同じ物理ページを解放していた。
    - その結果、task struct や kernel stack に再利用されるページが壊れ、`/bin/pwd` や `wait4()` の直後に parent/child のメタデータが破壊されていた。
    - なお、この欠陥は今回の作業で新規に入ったものではなく、**初期実装時点から潜在していた設計上の問題**だった。
        - 初期コミットの時点で `task_fork()` は FD を浅くコピーしており、`FT_DIR` はすでに `close()` で `pmm_free()` される実装だった。
        - 後に RetroFS を導入した際、同じ所有権モデルのまま `FT_RETROFS` が追加され、同型の問題が拡大した。
        - これまでは表面化しにくかったが、musl BusyBox の `ash` が `fork/exec/wait/close` を高頻度に踏むことで顕在化した。
- **修正内容**:
    - [kernel/fs.c](/home/itoh/MyOS/Github-Orthox-64/kernel/fs.c) に `fs_clone_fd()` / `fs_release_fd()` を追加。
    - `FT_PIPE` は refcount、`FT_SOCKET` は既存の duplicate 経路、`FT_DIR` / `FT_RETROFS` は物理ページを新規確保して深いコピーに統一。
    - [kernel/task.c](/home/itoh/MyOS/Github-Orthox-64/kernel/task.c) の `task_fork()` を `fs_clone_fd()` ベースへ変更し、途中失敗時の child 側 cleanup も追加。
    - [kernel/fs.c](/home/itoh/MyOS/Github-Orthox-64/kernel/fs.c) の `sys_dup2()` も同じ helper を使うように変更し、同一プロセス内の FD 複製でも二重解放が起きないようにした。

### 2.2 PMM と syscall 互換の補強
- **PMM の競合対策**:
    - [kernel/pmm.c](/home/itoh/MyOS/Github-Orthox-64/kernel/pmm.c) に spinlock を導入し、`pmm_alloc()` / `pmm_free()` / `pmm_incref()` / `pmm_get_ref()` の更新を保護した。
- **不足 syscall の実装**:
    - [kernel/syscall.c](/home/itoh/MyOS/Github-Orthox-64/kernel/syscall.c) に `sysinfo(99)`, `getuid(102)`, `getgid(104)`, `geteuid(107)`, `getegid(108)`, `getppid(110)` を追加。
    - `sysinfo.totalram` は Limine memmap から計算する最小実装とした。
- **`getcwd()` の Linux 互換化**:
    - [kernel/fs.c](/home/itoh/MyOS/Github-Orthox-64/kernel/fs.c) の `sys_getcwd()` を、成功時に `0` ではなく文字数+1 を返す形へ修正。

### 2.3 書込み系・`at_test` の互換修正
- **`rm` が対話待ちになる問題を修正**:
    - `faccessat()` が mode bits を機械的に見るだけで root 相当の特例を持たず、BusyBox 側から「書込み不可」と判定されていた。
    - [kernel/fs.c](/home/itoh/MyOS/Github-Orthox-64/kernel/fs.c) の `sys_faccessat()` を root 前提の単純な Linux 互換寄り挙動へ調整した。
- **`at_test` の mode mismatch を修正**:
    - RetroFS 上で `mkdirat()` / `openat(O_CREAT)` したファイル・ディレクトリは、作成後の `stat` で `0555/0444` に見えていた。
    - 原因は RetroFS の作成系が mode を保持しておらず、`retrofs_stat_path()` が固定 mode を返していたこと。
    - これを避けるため、作成系は writable overlay として RAMFS を優先する方針に変更。
    - [kernel/fs.c](/home/itoh/MyOS/Github-Orthox-64/kernel/fs.c) の `sys_open(O_CREAT)` / `sys_mkdir()` で RetroFS 作成経路を通さず、RAMFS を優先するようにした。

### 2.4 デバッグログの整理
- 調査中に追加した `pwd` 専用トレース、PF/COW 詳細ログ、PMM アドレス追跡ログ、`fork/zombie/reap` の追加ログを削除。
- 例外ログは通常運用向けに戻しつつ、fatal exception 自体は引き続き追える状態を維持した。

## 4. 最終ステータス
- musl libc への一本化後に残っていた BusyBox smoke の阻害要因は解消済み。
- 現時点で、`ash` 経由の外部コマンド実行、FD 継承、基本的なファイル作成・削除、`at_*` 系 API は実用上成立している。

## 5. 今後の課題
- RetroFS 自体が mode を永続化していないため、将来的には writable overlay 依存ではなく、FS レイヤとして mode/metadata を一貫保持する設計に寄せる余地がある。
- 現在の `sys_faccessat()` は root 前提の単純化実装なので、uid/gid を導入する段階でアクセス判定を再設計する必要がある。
- `muslbusyboxsmoke` は通ったため、次はより広いユーザーランド群や並列負荷を伴うテストで regressions を点検する。

## 6. 今後の改修プラン

### 6.1 方針
- 一気に `struct file` 相当の層へ全面移行するのはリスクが高いため、**小さな操作単位で段階的に改修する**。
- まず不要な TAR 系コードを削除して分岐を減らし、その後に FD 所有権の整理、最後に `struct file` 導入へ進む。

### 6.2 Phase 0: TAR 系コードの削除
- **目的**:
    - すでに利用価値のない TAR 経路を削除し、FS 分岐を減らす。
    - 以後の `struct file` 化で対象 backend を絞り込む。
- **対象**:
    - `FT_TAR`
    - `module_request` 経由の `.tar` 探索
    - rootfs.tar 前提の lookup/stat/dirents 構築コード
- **想定工数**:
    - 削除のみで 0.5〜1日
    - テストと周辺修正込みで 1〜2日

### 6.3 Phase 1: 現行 FD 所有権の共通化
- **目的**:
    - 現在の設計を維持したまま、`fork/dup/close` の lifetime バグを起こさない状態に固定する。
    - これは `struct file` の導入ではなく、現行 `file_descriptor_t` 設計の安全化フェーズ。
- **内容**:
    - `fs_clone_fd()` / `fs_release_fd()` を正式な共通入口として整理する。
    - 全 `FT_*` について clone/release 方針を明文化する。
    - `fork`, `dup2`, `close`, `FD_CLOEXEC` 経路を統一的に点検する。
- **想定工数**:
    - 0.5〜1.5日

### 6.4 Phase 2: 読み取り系 backend から `struct file` を導入
- **目的**:
    - 破壊リスクの低い backend から `struct file` 相当の open file object 層を導入する。
- **対象候補**:
    - `DIR`
    - `RETROFS`
    - `CONSOLE`
    - 必要なら `USB`
- **内容**:
    - `task->fds[]` を「FD 番号 -> file object 参照」に近づける。
    - `struct file` に `refcount`, `offset`, `status flags`, `ops`, `private_data` を持たせる。
    - `read`, `stat`, `lseek`, `getdents`, `release` を先に移行する。
- **想定工数**:
    - 2〜4日

### 6.5 Phase 3: 書き込み系・特殊系 backend の移行
- **目的**:
    - 最後に `RAMFS`, `PIPE`, `SOCKET` まで `struct file` ベースへ統一する。
- **内容**:
    - `write`, `truncate`, `pipe`, `dup`, `fcntl` などの経路を移行する。
    - backend ごとの差異を `file_ops` と backend object 側へ押し込む。
- **想定工数**:
    - 3〜5日

### 6.6 実施順の結論
1. TAR 系コードを削除する。
2. Phase 1 で現行 FD 所有権を共通化する。
3. Phase 2 で `DIR` / `RETROFS` から `struct file` を導入する。
4. Phase 3 で `RAMFS` / `PIPE` / `SOCKET` を移行する。

### 6.7 備考
- `struct file` 相当の層は現時点では未実装であり、現在は `file_descriptor_t` が FD エントリと open file state を兼ねている。
- 今回の欠陥は、この OS 的な抽象層の不足が `fork()` 後の二重解放として顕在化したものと整理できる。
