## Scheduler Regression and DOOM
- DOOM slowdown root cause was not musl. It was the scheduler policy introduced with timer-driven rescheduling.
- The previous timer IRQ path called `task_request_resched()` on every LAPIC timer tick. With a 10ms tick, this caused excessive preemption and made DOOM behave as if time were jumping forward.
- The fix was to switch to a minimal timeslice model:
  - add `timeslice_ticks` to each task
  - decrement it on each timer tick
  - request reschedule only when the slice expires
  - current setting: `TASK_TIMESLICE_TICKS = 5`
- After this change, DOOM performance returned to the earlier fast behavior without removing the scheduler entirely.
- The temporary profiling code inserted into DOOM for investigation was removed again.
- The `w_file_stdc.c` `mmap`-based WAD read optimization was kept.

## DOOM Remaining Work
- BGM and sound quality fixes are on hold for now.
- Next DOOM work should start from the faster scheduler baseline, not from the old always-reschedule path.

## 本日の最終到達点と成果

### 1. 初回ユーザタスク起動バグの修正
- **問題**: `Starting first user task...` 直後に Page Fault (#PF) が発生していた。
- **原因**: 初回タスク起動時に Linux/musl ABI が期待する初期スタック (`argc`, `argv`, `envp`, `auxv`) が構築されておらず、未マップ領域を参照していた。
- **対策**: `task_execve` からスタック構築ロジックを `task_prepare_initial_user_stack` として共通化し、`kernel/main.c` の初回起動時にも適用。また `auxv` の構築順序（valueを積んでからtypeを積む）を修正。
- **結果**: 初回シェルが正常に起動し、`Welcome to OrthOS Shell!` を確認。

### 2. USB ルート環境での musl 動作確認と不具合修正
- **問題**: `usbmuslcheck` 実行時に `/bin-musl/ash` が見つからない、または巨大ファイル読み込み中に USB 通信がハングアップしていた。
- **原因と対策**:
  - **xHCI 通信ハング**: 連続した Bulk 転送によるコントローラの輻輳。`usb_msc_bot_command` 完了後に 1ms の遅延を入れることで解消。
  - **TAR パース崩壊**: ハードリンク (`typeflag='1'`) を実データありと誤認してシークしていたバグを修正。また macOS APFS での `cp` クローン問題を回避するため、`cat` による実体コピーへ変更。
  - **LFN バグ**: ファイル名が13文字の倍数の場合に NUL 終端エントリが欠落するスクリプトバグを修正。
  - **パフォーマンス**: TAR 探索時の FAT チェーン辿りをキャッシュ化し $O(1)$ に改善。
- **結果**: **`usbmuslcheck: PASS`** を確認。USB 上の 84MB の `rootfs.tar` から musl バイナリが完全に動作することを実証。

## **追加の作業：キー入力の挙動改善とビルド復旧**

### 1. キー入力のエコーバックとバックスペースの修正
- **問題**: 
  - キーを入力すると同じ文字が2文字表示される（二重エコー）。
  - バックスペースキー（BS/DEL）を押しても画面上の文字が消えず、入力位置も戻らない。
- **原因**: 
  - **二重エコー**: `user/sh.c` とカーネルの `sys_read` の両方でエコーバックを行っていた。
  - **バックスペース**: カーネル側でバックスペース文字 (`\b`) を単なる文字としてエコーしており、画面操作用の制御文字（`\b \b`）を送信していなかった。またシェル側でも表示消去の処理が欠落していた。
- **対策**:
  - **二重エコー解消**: `user/sh.c` の `get_line` 内での `write(1, &c, 1)` を削除。エコーバックをカーネル側の一箇所に集約。
  - **バックスペース改善**: カーネルの `sys_read` (in `kernel/fs.c`) において、`\b` または `0x7F` (DEL) を受け取った際、エコーバックとして `\b \b` (一歩戻る、空白で消す、一歩戻る) を送信するように修正。
- **結果**: 1キー入力で1文字のみ表示され、バックスペースで正しく画面上の文字を消去できるようになった。

### 2. ビルドエラー解消と QEMU 起動の復旧
- **問題**: 
  - `make all` 実行時に `user/ttytest.c` や `user/sigmasktest.c` で musl ヘッダとの不一致によるビルドエラーが発生。
  - 依存関係の不足により `orthos.iso` が正しく生成されず、QEMU が起動不能になった。
- **対策**:
  - **ttytest.c 修正**: musl の `struct termios` の仕様に合わせ、`c_ispeed`/`c_ospeed` を `__c_ispeed`/`__c_ospeed` へ修正。
  - **Makefile 整理**: `mk/user-musl.mk` に汎用ビルドルールを追加。また、`iso/limine.conf` と `Makefile` の依存関係を、確実にビルドが通る最小構成（`kernel.elf`, `sh.elf`, `rootfs.tar`）に一時的に絞り、起動を優先。
- **結果**: `orthos.iso` の正常なビルドと、QEMU の起動復旧を確認。
