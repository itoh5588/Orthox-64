# 日報: musl/BusyBox 検証の整理と RetroFS 優先への方針転換 (Day 24)
**日付: 2026年4月11日**

## 1. 本日の目標
- `v1` の再確認結果を踏まえ、現行 kernel のどこが壊れているかを絞る。
- `tar` root と `RetroFS` root を切り分け、どちらを本命の検証対象にすべきか整理する。
- `muslcheck` がどこで失敗しているかを syscall 単位で潰す。

## 2. 実施内容

### `v1` 基準との比較を進めた
- `v1.0.0` 基準では `musl + BusyBox` の `exec` 自体は通っていた。
- これにより、現行 tree 側の問題であることが再確認できた。
- 同時に、`v1` 再現には当時の ports/toolchain 依存があり、完全再演のコストが高いことも分かった。

### syscall exit trace による戻り値破壊を修正した
- [`kernel/syscall_entry.S`](/home/itoh/MyOS/Github-Orthox-64/kernel/syscall_entry.S)
  - 一時 trace により `%rax` が壊れていたため、trace 呼び出しを外して `v1` に近い形へ戻した。
- [`kernel/syscall.c`](/home/itoh/MyOS/Github-Orthox-64/kernel/syscall.c)
  - 一時 trace 群を整理し、通常の syscall 復帰を優先した。
- この結果、`tar-only` boot で見えていた `/etc/bootcmd` の `empty read` は解消した。

### `execve` 周辺を `v1` 寄せで整理した
- [`kernel/task.c`](/home/itoh/MyOS/Github-Orthox-64/kernel/task.c)
  - `task_execve()` 周辺の一時回避や過剰な debug を落とし、`v1` に近い流れへ戻した。
  - ただし `ash -c` の長い文字列を扱うため、`EXEC_MAX_VEC_STR_LEN=512` と `EXEC_COPY_PAGES=10` は維持した。
- これにより、以前のような `#PF(User)` ではなく、userland 側の通常失敗に変化した。

### `ash` 経由の `ls /bin` 失敗を `lstat` 未実装まで絞った
- [`include/syscall.h`](/home/itoh/MyOS/Github-Orthox-64/include/syscall.h)
  - `SYS_LSTAT` を追加した。
- [`kernel/syscall.c`](/home/itoh/MyOS/Github-Orthox-64/kernel/syscall.c)
  - `SYS_LSTAT` を `sys_stat()` へ接続した。
  - 一時的に `pid 4` 向けの syscall trace と `lstat` path trace を入れて、BusyBox `ls` の失敗 syscall を観測した。
- その結果、
  - `ash` は `/bin/ls /bin` を正しく `exec` している
  - BusyBox `ls` は `/bin/*` に対して `lstat()` を大量に呼ぶ
  - そこで失敗して `Operation not permitted` になっていた
 ことを確認した。
- `lstat` 実装後、`ls /bin` 自体は通るようになった。

### `muslcheck` の失敗を `touch` と `cp` の errno 問題まで縮めた
- `muslcheck` の中身は [`user/sh.c`](/home/itoh/MyOS/Github-Orthox-64/user/sh.c) の
  `pwd && ls /bin && touch /muslcheck.tmp && mkdir /muslcheck.dir && cp ...`
  の列である。
- まず `touch /muslcheck.tmp` が失敗していたため、
  [`kernel/fs.c`](/home/itoh/MyOS/Github-Orthox-64/kernel/fs.c)
  に `sys_utimensat()` を追加し、
  [`kernel/syscall.c`](/home/itoh/MyOS/Github-Orthox-64/kernel/syscall.c)
  から `SYS_UTIMENSAT` を接続した。
- これで `touch` と `mkdir` は通るようになった。
- 続いて `cp /hello.txt /muslcheck.dir/hello.copy` の途中で
  `can't stat '/muslcheck.dir/hello.copy': Operation not permitted`
  が出ることを確認した。
- これは path lookup そのものではなく、`stat/lstat/fstatat` が missing file に対して `-ENOENT` ではなく単なる `-1` を返し、musl/BusyBox 側で `EPERM` 相当に見えていた可能性が高い。
- そのため [`kernel/syscall.c`](/home/itoh/MyOS/Github-Orthox-64/kernel/syscall.c) で
  `SYS_STAT`, `SYS_LSTAT`, `SYS_FSTATAT` を `-ENOENT` 返しへ揃える修正を入れた。
- ただし、この最後の再検証は途中で打ち切っており、通し確認は未完了。

### `tar` root は control に留め、`RetroFS` を本命にする方針へ切り替えた
- `tar-only` root での検証により、
  - `fork/exec` 崩壊の切り分け
  - `lstat`
  - `utimensat`
  - `errno`
  のような足りない syscall 面は見えてきた。
- 一方で `tar` root は writable semantics や errno の整合が不足しており、本番 root としては不適切であることも明確になった。
- 現行 kernel には `RetroFS` の boot 統合自体は入っているため、今後の本命検証は `rootfs.img` を使う通常 boot に戻す方針とした。

## 3. 本日確認できたこと
- `v1` では `musl + BusyBox` 実行は通っていた。
- 現行 tree では一時 trace や回避コードが混ざり、切り分けを難しくしていた。
- `ash` の `exec` 自体は通っており、問題はその後の syscall 実装不足と errno の粗さに移っている。
- `ls /bin` は `lstat` 実装追加で通るようになった。
- `touch` は `utimensat` 実装追加で通るようになった。
- `muslcheck` の残件は `cp` 以降、特に missing file に対する `stat/lstat/fstatat` の errno 整合に縮んでいる。
- ただし、`tar` root でこの整合を詰めても、本命の runtime 品質保証には足りない。

## 4. 現在の問題点

### 1. `tar` root は検証用 control であり、本命 root ではない
- writable な root filesystem としての挙動が十分でない。
- errno や作成系 syscall の意味論も未整備な点がある。
- よって、ここを完全に作り込むのは優先順位が低い。

### 2. `RetroFS` root での通し再検証が未完了
- 現行 kernel には `RetroFS` boot 統合が入っている。
- しかし、`muslcheck` を `RetroFS root` 上で再度通して、現在の残件がそのまま再現するかはまだ確認しきれていない。

### 3. debug trace がまだ残っている
- [`kernel/task.c`](/home/itoh/MyOS/Github-Orthox-64/kernel/task.c)
  - `/bin/*` 向けの一時 `exec` trace が残っている。
- [`kernel/syscall.c`](/home/itoh/MyOS/Github-Orthox-64/kernel/syscall.c)
  - `pid 4` 向け syscall trace と `lstat` trace が残っている。
- [`kernel/fs.c`](/home/itoh/MyOS/Github-Orthox-64/kernel/fs.c)
  - `/bin/*` 周辺の一時 fs trace が残っている。
- `RetroFS` 側の検証が済んだら、不要な trace は落とす必要がある。

## 5. 次の優先タスク
1. `rootfs.img` を載せた通常の `orthos.iso` で起動し、`RetroFS root` 上で `muslcheck` を直接走らせる。
2. `tar-only` で入れた修正のうち、`RetroFS` 側でも必要なものだけを残す。
3. `RetroFS root` 上で `touch`, `mkdir`, `cp`, `stat`, `rm`, `rmdir` が通るかを確認する。
4. `SYS_STAT`, `SYS_LSTAT`, `SYS_FSTATAT` の `-ENOENT` 返し修正が正しいかを再検証する。
5. 問題が `RetroFS` では再現しないなら、`tar` 固有の制限として切り離す。
6. 残る一時 trace を整理して、kernel を通常状態へ戻す。

## 6. 本日の結論
- `tar` root の検証で、`musl + BusyBox` の破綻はかなり小さくできた。
- ただし `tar` は機能不足が多く、本命の root filesystem には向かない。
- 今後の主戦場は `RetroFS root` であり、`rootfs.img` を使う通常 boot に戻して詰めるべきである。
