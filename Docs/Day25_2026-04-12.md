### まず既存の bootcmd ベース再現で症状を観測し直した
- [`tests/busybox_stat_missing_smoke.sh`](/home/itoh/MyOS/Github-Orthox-64/tests/busybox_stat_missing_smoke.sh)
  - `orthos.iso + rootfs.img` の通常 boot で `/bin/ash` から `stat /definitely-missing` を打つ専用 smoke を用意した。
  - 当初の再現では、過去ログにあった
    - `Vector: 000000000000000D`
    - `RIP: 0xFFFFFFFF800025D2`
    が `vmm_free_user_pml4()` に一致することを確認した。
- これにより、BusyBox `stat` のユーザー空間エラーパスの裏で、アドレス空間破棄タイミングがおかしい可能性が強くなった。

### `task_execve()` の旧 CR3 即時解放をやめ、遅延解放へ変更した
- [`include/task.h`](/home/itoh/MyOS/Github-Orthox-64/include/task.h)
  - `struct task` に `deferred_cr3` を追加した。
- [`kernel/task.c`](/home/itoh/MyOS/Github-Orthox-64/kernel/task.c)
  - `task_execve()` で新しいユーザーアドレス空間へ切り替えた直後に旧 `cr3` を `vmm_free_user_pml4()` へ渡していた処理をやめた。
  - 代わりに旧 `cr3` を `t->deferred_cr3` に退避し、`task_reap_locked()` で task 終了時に解放する流れへ変更した。
- この変更で、少なくとも「exec 直後の page-table teardown で kernel 例外になる」系の壊れ方は止まった。

### shell/bootcmd ノイズを避ける focused regression を追加した
- [`user/staterrno.c`](/home/itoh/MyOS/Github-Orthox-64/user/staterrno.c)
  - `stat("/definitely-missing")` を直接呼び、`rc` と `errno` を出す最小プローブを追加した。
- [`user/muslstatmissingdriver.c`](/home/itoh/MyOS/Github-Orthox-64/user/muslstatmissingdriver.c)
  - `fork -> exec(/bin/staterrno.elf) -> waitpid` を first user task として回す専用ドライバを追加した。
- [`tests/musl_statmissing_smoke.sh`](/home/itoh/MyOS/Github-Orthox-64/tests/musl_statmissing_smoke.sh)
  - 上記ドライバを使い、plain `stat()` の missing-path エラーパスだけを focused に検証する smoke を追加した。
- この結果、
  - `staterrno:start`
  - `staterrno: rc=-1 errno=2`
  - `staterrno:end`
  - `muslstatmissingdriver:status=0`
  を確認し、plain `stat()` 自体は正常であることを証明できた。

### BusyBox `stat` 自体も focused smoke で再検証した
- [`user/muslbusyboxstatdriver.c`](/home/itoh/MyOS/Github-Orthox-64/user/muslbusyboxstatdriver.c)
  - `fork -> exec(/bin/stat /definitely-missing) -> waitpid` を first user task として実行するドライバを追加した。
- [`tests/musl_busybox_stat_missing_smoke.sh`](/home/itoh/MyOS/Github-Orthox-64/tests/musl_busybox_stat_missing_smoke.sh)
  - BusyBox `stat` applet の missing-path エラーパスだけを focused に検証する smoke を追加した。
- 修正後の focused smoke では、

- これにより、BusyBox `stat` の missing-path error exit 自体は正常であり、「別 issue の正体は BusyBox 固有バグではなく `execve` 側の古い CR3 解放タイミングだった」と結論づけられた。

## 3. 本日確認できたこと
- shell 本体の `stat("/non-existent-file-xyz")` は以前から `ret=-1 errno=2` を返しており、`stat` syscall 自体は少なくとも基本経路で正常だった。
- 過去の kernel exception は `vmm_free_user_pml4()` 内で起きており、BusyBox `stat` の userland エラー処理そのものが本体ではなかった。
- `task_execve()` で旧ユーザーアドレス空間を即時 free していたのが危険で、これを task 終了時の遅延解放へ変えると focused repro は正常化した。
- `fork -> exec(/bin/staterrno.elf)` は `errno=2` で正常終了する。
- `fork -> exec(/bin/stat /definitely-missing)` も `No such file or directory` を出して `waitpid()` で回収できる。
- したがって、Day24 時点で見えていた「BusyBox stat missing-path issue」は、今日の時点では separate issue としてほぼ収束している。

## 7. 本日の結論
- Day24 から持ち越した BusyBox `stat` missing-path 問題は、BusyBox applet 固有の不具合ではなかった。
- 本体は `task_execve()` における旧ユーザー CR3 の即時解放であり、これが userland の通常エラーパスを kernel exception に変えていた。
- `deferred_cr3` による遅延解放へ変えたことで、focused regression では plain `stat()` も BusyBox `stat` も正常な missing-path error exit に戻った。
- よって、この separate issue は「根因特定、修正、focused regression 追加」まで完了したと判断できる。

## 8. 改善点

### 1. bootcmd ベースの統合検証はノイズが多い
- `muslcheck` や shell 起動、bootcmd、既存の debug 出力が混ざるため、切り分けに不向きだった。
- 今回のように first user task を専用 driver に差し替える focused smoke の方が、再現性と診断効率が高い。

### 2. `execve`/address-space lifetime の回帰テストを増やすべき
- 今回のバグは「実装は動いて見えるが、エラーパスでだけ壊れる」種類だった。
- `fork -> exec -> missing-path error -> waitpid` のような負経路を標準 smoke に含めるべきである。

### 3. issue 切り出しを早めた方が診断効率が高い
- Day24 では `muslcheck` 本体の一部として見えていたが、実際には別問題だった。
- shell、BusyBox applet、kernel exec、filesystem errno を同時に追うより、今日のように issue 化して focused に潰した方が早い。

### 4. serial evidence を前提にしたドライバ型テストは継続利用すべき
- 今回追加した `muslstatmissingdriver` / `muslbusyboxstatdriver` は、そのまま回帰テスト資産になる。
- 今後も `fork/exec/wait`, `errno`, `signal`, `pipe`, `dirent` などは同様の focused driver に分解していく価値がある。
