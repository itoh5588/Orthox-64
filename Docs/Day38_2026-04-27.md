# 日報: RAMFS 修正の上位 self compile 確認と write errno 変換修正
**日付: 2026年4月27日**

## 1. 本日の目標
- Day 39 で修正した RAMFS 1 MiB 化と full 時 `-EFBIG` 返却が、上位の self compile 経路でも効いていることを確認する。
- full `sh.c` の native compile / stage2 smoke を再実行し、`cc1 -> as -> ld` まで完走するか確認する。
- `write()` が 0 バイト成功を返す、またはエラーを userland に正しく伝えない経路が残っていないか確認する。

## 2. 重要な結論
- full `sh.c` の stage2 compile は PASS した。
- `native_stage2_userland_smoke.sh` は元々 `sh_stage2` ルールを持っていたが、`all` ターゲットに含まれていなかった。
- そのため、これまでの stage2 smoke は `sh_stage2` を実際には検証していなかった。
- `sh_stage2` を `all` に追加し、`gcc_stage2` で full `sh.c` を `cc1 -> as -> ld` まで通す検証に拡張した。
- RAMFS full 時の kernel 側 `write()` は `-EFBIG` を返せていたが、userland の `_write()` wrapper が errno 変換をしていなかった。
- その結果、userland からは `write()` が `-27` のような負値をそのまま返し、`errno` は 0 のままになる問題があった。
- `_write()` / `_read()` / `_close()` を `orth_status_ret()` 経由に変更し、`writev()` / `readv()` wrapper も追加した。

## 4. 追加で見つかった問題

### 4.1 RAMFS full 時の userland errno 変換漏れ
`write()` が 0 を返す path の確認として、RAMFS full 条件を userland test に追加した。

追加した検証内容:
- `/work/ramfs_full.bin` を作成。
- 4 KiB block を 256 回書き、1 MiB まで埋める。
- さらに 1 byte の `write()` を行い、`-1` / `errno == EFBIG` になることを確認する。
- 同じ条件で `writev()` も `-1` / `errno == EFBIG` になることを確認する。

最初の実行結果:
```text
ftruncsave_test: FAIL: full RAMFS write should set EFBIG errno=0
```

この時点で分かったこと:
- kernel 側 RAMFS は full 時に `-EFBIG` を返す実装になっていた。
- しかし userland の `_write()` が `orth_status_ret()` を通しておらず、負 errno をそのまま返していた。
- そのため、C library の `write()` として期待される `-1` / `errno=EFBIG` になっていなかった。

### 4.2 wrapper 側の問題箇所
対象ファイル:
- `user/syscalls.c`

変更前:
```c
ssize_t _write(int fd, const void *buf, size_t count) {
    return (ssize_t)orth_syscall3(SYS_WRITE, ...);
}
```

この形では、kernel が `-EFBIG` を返しても userland では `-27` がそのまま見える。
POSIX 互換の syscall wrapper としては、`-1` を返し `errno` に 27 を設定する必要がある。

## 5. 修正内容

### 5.1 `_write()` / `_read()` / `_close()` の errno 変換
`user/syscalls.c` の以下を `orth_status_ret()` 経由に変更した。

- `_write()`
- `_read()`
- `_close()`

変更後:
```c
ssize_t _write(int fd, const void *buf, size_t count) {
    return (ssize_t)orth_status_ret(
        orth_syscall3(SYS_WRITE, (uint64_t)fd, (uint64_t)buf, (uint64_t)count)
    );
}
```

これにより、kernel から `-EFBIG`、`-EBADF`、`-EINVAL` などが返った場合に userland の `errno` へ反映される。

### 5.2 `writev()` / `readv()` wrapper の追加
`user/syscalls.c` に `writev()` / `readv()` を追加した。

```c
ssize_t writev(int fd, const struct iovec *iov, int iovcnt) {
    return (ssize_t)orth_status_ret(
        orth_syscall3(SYS_WRITEV, (uint64_t)fd, (uint64_t)iov, (uint64_t)iovcnt)
    );
}

ssize_t readv(int fd, const struct iovec *iov, int iovcnt) {
    return (ssize_t)orth_status_ret(
        orth_syscall3(SYS_READV, (uint64_t)fd, (uint64_t)iov, (uint64_t)iovcnt)
    );
}
```

`cc1` は `writev()` を使うため、RAMFS full 時の errno 伝播確認として重要。

### 5.3 RAMFS full 回帰テストの追加
`user/ftruncsave_test.c` に `check_ramfs_full_write()` を追加した。

確認する内容:
- RAMFS 上の 1 ファイル 1 MiB 上限まで書ける。
- 上限到達後の追加 `write()` は `-1` / `EFBIG`。
- 上限到達後の追加 `writev()` も `-1` / `EFBIG`。
- テスト後は `/work/ramfs_full.bin` を unlink する。

既存の ftruncate / save smoke にこの検証を統合したため、RAMFS 保存系の回帰と一緒に確認できる。

対象ファイル:
- `user/ftruncsave_test.c`

## 8. 今日の評価
- Day 39 の RAMFS 修正は、単体 RAMFS probe だけでなく full `sh.c` の stage2 compile にも有効だった。
- `native_stage2_userland_smoke.sh` の検証範囲に穴があり、`sh_stage2` ルールが存在するだけで実際には走っていなかった点を修正できた。
- RAMFS full 時の kernel semantics だけでなく、userland の errno 変換まで確認できる回帰テストを追加できた。
- `cc1` のように `writev()` を使う workload では、kernel の返り値だけでなく libc / syscall wrapper の変換も重要であることが分かった。

## 9. 次回への引き継ぎ
1. `user/syscalls.c` は今回の差分以前から大きく変更されているため、必要なら別途整理して責務や weak symbol の扱いを確認する。
2. `write()` / `writev()` 以外にも、負 errno をそのまま返している wrapper が残っていないか `user/syscalls.c` 全体を点検する。
3. `native_stage2_userland_smoke.sh` は `sh_stage2` を含むようになったため、今後の self compile 回帰ではこの smoke を優先的に使える。
4. RAMFS の 1 MiB 固定上限は当面維持でよいが、将来的な可変サイズ化または page chain 化は引き続き検討する。
5. `ORTHOX_MEM_PROGRESS` は診断用として残し、通常 build では無効のままでよい。

## 10. メモ
- `write()` が 0 を返す問題そのものは RAMFS 側で修正済みだったが、今回は userland で errno が立たない別の問題が露出した。
- `cc1` 長時間化の再発防止には、RAMFS full の `write()` と `writev()` の両方を明示的にテストするのが有効。
- stage2 smoke は「ルールがある」だけでは不十分で、`all` に含まれているか、ログ検証で該当コマンドを確認しているかまで見る必要がある。
