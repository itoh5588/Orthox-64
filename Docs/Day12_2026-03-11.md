# Day12 (2026-03-11)

## 今日の到達点

今日は OrthOS 上で、ツールチェーンの最小通し実行を確認した。

実際に通した手順は以下のとおり。

1. `cc1.elf -nostdinc hello.c -o hello.s`
2. `as hello.s -o hello.o`
3. `ld -m elf_x86_64 -e _start crt0.o hello.o syscalls.o libc.a -o hello.elf`
4. `hello.elf`

最終的に OrthOS シェル上で `Hello, world` を出力できた。

これは単なる `hello world` ではなく、既存 OSS の移植基盤が OS 上で自己完結して動くことを示す重要な確認である。

- `cc1` が C ソースを読んでアセンブリへ変換できる
- `as` がオブジェクト生成できる
- `ld` が実行 ELF を生成できる
- 生成した ELF を OrthOS がその場で実行できる

この流れが通ったことで、OrthOS は「DOOM が動く OS」であるだけでなく、「既存ツールチェーンを OS 内で使ってユーザー空間プログラムを生成し、そのまま実行できる OS」に一歩進んだ。

### 2. GNU as / ld が使う file-backed mmap への対応

既存の `mmap` 実装は `MAP_ANONYMOUS` 前提であり、GNU `as` が入力ファイルを `mmap(fd, ...)` で扱う経路で停止していた。

そこで、最小限の file-backed `mmap` を実装した。

- 匿名 mmap は従来通り利用可能
- `fd` 付き `mmap` では、対応するファイル内容を新規ページへコピーしてユーザー空間へマップ
- 少なくとも `as` / `ld` が必要とする読み取り用途に対応

## 今日わかったこと

### 1. OrthOS は既存ツール移植に十分な段階へ近づいている

`newlib` と最小 syscall 層、ELF 実行、rootfs、書き込み可能ファイル、`mmap` が揃ってくると、既存の GNU 系ツールがかなりそのまま動く。

これは今後の移植性に直結する。

### 2. DOOM が動くだけでは見えない欠陥が、ツールチェーンで露出する

DOOM は大規模実アプリとして重要だが、`cc1/as/ld` のようなツールはファイル I/O、`mmap`、小さな POSIX 互換性の不足を鋭く突いてくる。

つまり、DOOM とツールチェーンの両方を回すことで、OS の成熟度を異なる角度から検証できる。

### 3. 「OS 上でコンパイルして実行できる」ことの意味は大きい

OrthOS の価値は独自 OS であること自体ではなく、既存資産をどれだけ早く動かせるかにある。

今回 `cc1 -> as -> ld -> 実行` が通ったのは、その方向性が正しいことの証明になった。

DOOM が動き、音まで出せていることに加えて、開発用ツールまで OS 上で回り始めたのは大きな前進である。

## 翌日追補: `stat bin/sh` と USB root `/bin/sh` 起動の解決

翌日の切り分けで、前日の「高い LBA の `READ(10)` が壊れている」という結論は不正確だったことが分かった。

### 実際の原因

- `fs_try_usb_root_tar_lookup()` 自体は `./bin/sh` の tar header を正しく見つけていた
- しかし `sys_stat()` が metadata 参照だけで済む場面でも `fs_try_active_root_lookup()` に `data` バッファを要求していた
- そのため `stat bin/sh` が、サイズ確認のつもりで `bin/sh` 本体 369,360 bytes を USB から丸ごと読み始めていた
- さらに `usb_read_blocks_safe()` が 1-sector 読みを基本としていたため、必要以上に BOT command 数が増え、途中で `bulk OUT cc=0x6` を誘発していた

### 実施した修正

- `kernel/fs.c`
  - `sys_stat()` を metadata-only 化し、active root lookup に `data=NULL` を渡すよう修正
  - `usb_read_blocks_safe()` を「まず `count` まとめ読み、失敗したら 1-sector fallback」に変更

### 修正後の確認

- `mount usbroot /usb/rootfs.tar`
  - 成功
- `stat bin/sh`
  - 成功
- `cat hello.txt`
  - 成功
- `/bin/sh`
  - USB root 上の `bin/sh` を `execve()` で起動できた

### この時点での意味

OrthOS は

- USB MSC/BOT
- FAT 参照
- `/usb/...` 疑似マウント
- USB 上の `rootfs.tar` への root 切替
- その root 上の `/bin/sh` 起動

まで到達した。

## Boot 構成メモ (`iso/limine.conf`)

OrthOS の boot 時設定は [iso/limine.conf](/Users/itoh/orthOS-64/iso/limine.conf) に集約されている。

現状の役割は次の通り。

- `timeout: 3`
  - Limine メニューの待ち時間。
- `/OrthOS`
  - 起動エントリ名。
- `protocol: limine`
  - Limine ブートプロトコルを使用。
- `path: boot():/boot/kernel.elf`
  - カーネル本体の読み込み先。
- `module_path: boot():/boot/*.elf`
  - ユーザー空間プログラム群を boot module として先に読み込む。
  - 現在は `sh.elf`, `gcc.elf`, `cc1.elf`, `as.elf`, `ld.elf`, `doom.elf`, `mmaptest.elf`, `reaptest.elf`, `robusttest.elf` などを指定している。
- `module_path: boot():/boot/rootfs.tar`
  - 初期 rootfs を TAR の module として渡す。

この構成の意味は、OrthOS が現時点では

1. `kernel.elf` を Limine から直接起動し、
2. 必要な userland ELF と `rootfs.tar` を module として受け取り、
3. kernel 側でそれらを VFS / exec / shell に接続する

という流れで起動している、ということ。

したがって boot 時の挙動を変えたい場合、まず最初に確認すべき設定ファイルは `iso/limine.conf` である。
