# Day17 作業報告 (2026-03-21)

## 1. 今日の目的

- Day16 の末尾で整理した「SMP の wakeup / scheduling 側をさらに実経路へ降ろす」という方針に従い、今日は network socket 側の待機モデルを見直した。
- これまで console input と `wait4()` については
  - task を `TASK_SLEEPING` に落とす
  - イベント側から `task_wake()` する
という基本形を入れていた。
- 一方で socket 側は依然として
  - `recv`
  - `connect`
  - `accept`
  - `send`
の待機が `kernel_yield()` ベースの loop に残っており、SMP の wakeup モデルから遅れていた。
- 今日の主題は、この socket 側待機を同じ sleep/wake モデルへ寄せられるかを確認することに置いた。

## 2. 実装した内容

### 2.1 `net_socket_backend` に waiter を追加

- 対象ファイル:
  - `kernel/net_socket.c`
- `net_socket_backend` に以下の waiter を追加した。
  - `rx_waiter`
  - `tx_waiter`
  - `connect_waiter`
  - `accept_waiter`
- あわせて helper を追加した。
  - `socket_wake_waiter()`
  - `socket_set_waiter()`
  - `socket_clear_waiter()`

狙い:
- socket ごとに「今どの task が何待ちなのか」を backend 側に持たせる。
- 後段の lwIP callback や accept 完了経路から、その task を直接 `task_wake()` できる形へする。

### 2.2 receive 待機の sleep/wake 化

- `socket_recv_stream()` と `socket_recv_dgram()` の `while (!sock->rx_head)` loop を変更した。
- 具体的には、blocking mode の場合
  - current task を `TASK_SLEEPING` に落とす
  - `sock->rx_waiter = current`
  - `kernel_yield()`
  - 復帰後に waiter を clear
という流れにした。

これにより:
- 以前は単に CPU を譲るだけだった receive 待機が、
- 「この task は socket receive イベント待ち」という task state を伴う待機になった。

### 2.3 TCP send 待機の sleep/wake 化

- `socket_send_stream()` の以下 2 箇所を変更した。
  - `tcp_sndbuf() == 0` で送信 window が空のとき
  - `tcp_write()` が `ERR_MEM` を返したとき
- どちらも blocking mode の場合
  - task を `TASK_SLEEPING`
  - `sock->tx_waiter = current`
  - `kernel_yield()`
  - 復帰後に waiter clear
へ変更した。

目的:
- 送信可能になるイベントを lwIP callback 側から `task_wake()` できるようにする。

### 2.4 TCP connect 待機の sleep/wake 化

- `sys_connect()` 内の `while (sock->connecting)` loop を変更した。
- blocking connect の間、
  - task を `TASK_SLEEPING`
  - `sock->connect_waiter = current`
  - `kernel_yield()`
する形へ変更した。

これにより:
- connect 完了待ちも polling ではなく event-driven に寄せ始めた。

### 2.5 accept 待機の sleep/wake 化

- `sys_accept()` 内の `while (!listener->accept_head)` loop を変更した。
- blocking accept の場合
  - task を `TASK_SLEEPING`
  - `listener->accept_waiter = current`
  - `kernel_yield()`
とした。

意味:
- listen socket 側も「接続到着まで寝る」モデルに寄せた。

## 3. lwIP callback 側での wakeup 追加

### 3.1 UDP receive

- `udp_socket_recv()` で `queue_pbuf_data()` 成功後に
  - `socket_wake_waiter(&sock->rx_waiter)`
を呼ぶようにした。

これにより:
- UDP datagram 到着時に `recvfrom()` 待機 task を起こせる。

### 3.2 TCP stream receive / EOF / error

- `tcp_stream_recv()` で
  - payload queue 後
  - EOF (`!p`)
  - `ERR_MEM`
時に `rx_waiter` を wake するようにした。
- `tcp_stream_err()` では
  - `rx_waiter`
  - `tx_waiter`
  - `connect_waiter`
をまとめて wake するようにした。

意味:
- 受信完了、接続異常、EOF が待機 task にきちんと伝搬する。

### 3.3 TCP send completion

- `tcp_stream_sent()` で `tx_waiter` を wake するようにした。

意味:
- 送信バッファが進んだとき、send 側を polling で回さずに戻せる。

### 3.4 TCP client connect 完了

- `tcp_client_connected()` で
  - 成功時
  - エラー時
の両方で `connect_waiter` を wake するようにした。

意味:
- blocking connect が callback で復帰できる前提が整った。

### 3.5 listener accept 完了

- `tcp_listener_accept()` で child socket を accept queue に積んだあと、
  - `socket_wake_waiter(&listener->accept_waiter)`
を呼ぶようにした。

意味:
- accept 待ち task を接続到着イベントで即座に起こせる。

## 8. 追加作業: fork spread 復帰までの整理

### 8.1 runtime 切替と観測系の追加

- `fork` 子を親 CPU に固定した暫定策を剥がすため、まず spread/pinned を guest から切り替えられるようにした。
- 追加したもの:
  - `set_fork_spread()`
  - `get_fork_spread()`
  - `/bin/forkmode.elf`
  - `/bin/forkcputest.elf`
- これにより
  - pinned のまま基準線を取る
  - `forkmode on` で AP 分散を再開する
  - `forkcputest` で parent / child / reap の CPU 配置を見る
という診断が可能になった。

目的:
- `fork` 子を AP に散らしたときだけ壊れるのか、
- どの実経路で落ちるのか、
を shell から直接観測できるようにすること。

### 8.2 `syscall_entry` / direct return 周辺の補強

- `forkmode on` + `forkcputest 8` で、当初は
  - `syscall_entry` 冒頭の `mov %rsp, %gs:0x8`
で page fault が再現した。
- この経路を詰めるため、
  - `fork_child_entry`
  - `call_app`
で direct kernel-to-user return 前に `swapgs` を入れた。
- あわせて schedule 前に per-CPU の
  - `GS_BASE`
  - `KERNEL_GS_BASE`
を再装填するようにした。

意味:
- syscall return を経由しない user 復帰でも、次の `swapgs` が per-CPU 領域へ正しく着地する前提を強めた。
- AP 側の user task が syscall を打つ瞬間の GS 向きを安定化させるための措置である。

### 8.3 pipe を SMP-safe に修正

- AP 分散を戻すと、`forkcputest` 単体は通る一方で shell pipeline が hang するケースが残った。
- 調べると `pipe_t` は
  - `count`
  - `read_pos`
  - `write_pos`
  - `ref_count`
  - `read_waiter`
  - `write_waiter`
を lockless で触っており、reader / writer が別 CPU に散ると壊れうる状態だった。
- そこで `pipe_t` に `spinlock_t lock` を追加し、
  - `read`
  - `write`
  - `close`
  - `dup2`
  - `fork` 時の refcount 複製
を lock 下に揃えた。

結果:
- spread 有効のまま
  - `echo SMP | wc`
  - `printf 'a\nb\nc\n' | wc`
  - `ls /bin | head`
が通るようになった。
- つまり、fork spread を戻した時に残っていた実害の中心は pipe 側の SMP race だったと判断できる。

### 8.4 既定挙動を spread に復帰

- 上記の GS 周辺補強と pipe lock 化のあと、`forkmode on` に依存しなくても spread 既定での実行が通ることを確認した。
- 現在は `fork` 子の既定挙動を再び spread 側に戻している。

確認できたもの:
- shell pipeline
- `forkcputest`
- `pipestress`
- `reaptest`
- `smpstress`

ここでいう `smpstress` は、
  - `pipestress`
  - `forkcputest`
  - `reaptest`
を `execve()` 経由で反復する guest 内の soak test である。

## 9. timer / sleep / DNS 待機の整理

### 9.1 global tick が 4 CPU で 4 倍速になる問題

- SMP bring-up を進める中で、`lapic_timer_tick()` が全 CPU の local timer interrupt で `g_ticks_ms` を加算していることに気づいた。
- その結果、4 CPU 起動では `lapic_get_ticks_ms()` が実時間の約 4 倍速で進む。
- これは `sleep_ms()` や timer 依存処理全般に影響するため、ここで修正した。

### 9.2 global tick を `cpu0` のみで進めるよう修正

- `g_ticks_ms` は `cpu0` の timer tick だけで加算するように変更した。
- 一方で `g_ticks_ms_per_cpu[cpu_id]` は各 CPU の local timer で引き続き加算する。
- これにより、
  - 共通時間軸は 1 本
  - per-CPU local timer 観測は維持
という形に整理できた。

### 9.3 `sleep_ms()` を deadline wakeup に変更

- これまで `sleep_ms()` は `get_ticks_ms()` を見ながら `kernel_yield()` する polling loop だった。
- 今回 task に `sleep_until_ms` を追加し、
  - task を `TASK_SLEEPING`
  - `sleep_until_ms = now + ms`
  - timer tick 側で期限到達 task を `task_wake()`
という形へ変更した。
- wakeup 判定は `task_on_timer_tick()` で行い、global deadline の走査は `cpu0` 側のみで実施する。

### 9.4 DNS lookup 待ちも waiter 化

- `lwip_port_lookup_ipv4()` も `while (!g_dns_done) kernel_yield();` のままだったため、これも waiter 化した。
- 追加したのは
  - `g_dns_waiter`
  - `orthox_lwip_wake_dns_waiter()`
である。
- `dns_gethostbyname()` の callback `orthox_lwip_dns_found()` で waiter を wake するようにし、lookup 側では current task を `TASK_SLEEPING` にして待つ形へ変更した。

## 11. AP child `execve` 切り分けと初回 task switch 停止点の調査

### 11.1 背景

- 次の段階として、`cpu0` 固定を少し外し、`fork()` した child を started CPU 群へ round-robin で配置する実験を始めた。
- そのため `task_fork()` に `choose_spawn_cpu()` を追加し、`[smp] fork pid=... cpu=...` を serial に出すようにした。
- 最初の目的は、
  - `cpu1` に child を置けるか
  - その child が `execve()` と userspace 起動まで進めるか
を確認することである。

### 11.2 最初に取れたエビデンス

- 4 CPU 起動で `/bin/testtime.elf` を 2 回実行すると、
  - 1 回目: `[smp] fork pid=3 cpu=0`
  - 2 回目: `[smp] fork pid=4 cpu=1`
となり、2 回目は child が `cpu1` に振られていることを確認できた。
- この 2 回目では当初
  - `Exec: begin a=0 b=0 ...`
  - `Exec: alloc_user_stack failed`
  - `Exec: stack preparation failed`
  - `Vector: 0000000000000006`
が出た。
- ここで
  - `a` は `current->pid`
  - `b` は `current->cpu_affinity`
として埋めているため、
  - `fork pid=4 cpu=1`
なのに
  - `Exec: begin a=0 b=0`
になっているのは不一致である。
- この不一致は、「scheduler は child を `cpu1` に置こうとしているが、`task_execve()` が見ている current task は `pid=0` の idle task になっている」ことを示している。

### 11.3 `fork_child_entry` の `swapgs` は外したまま維持

- さらにその前段として、以前の `cpu1` child 実行では `syscall_entry` の
  - `mov %rsp, %gs:0x8`
で `#PF @ 0x8` が出ていた。
- これは `GS` ベース崩れのエビデンスだったため、`kernel/task_switch.S` の `fork_child_entry` から `swapgs` を削除した。
- この変更により、少なくとも「AP child が `syscall_entry` 冒頭で即死する」状態は抜けたため、以後この変更は維持している。

### 11.4 `execve()` の user pointer 直読みをやめた

- `task_execve()` はそれまで
  - `path`
  - `argv`
  - `envp`
を user pointer のまま kernel から直接たどっていた。
- これは SMP 切り分け以前に設計として脆いので、今回
  - `EXEC_COPY_PAGES`
  - `struct exec_copy_buf`
  - `copy_user_cstring()`
  - `copy_user_string_vector()`
を追加し、まず kernel 側 scratch buffer に文字列を吸い上げる形へ変更した。
- そのうえで
  - `fs_get_file_data(exec_copy->path, ...)`
  - `task_prepare_initial_user_stack(..., exec_copy->argv, exec_copy->envp)`
に渡すようにした。
- これにより、`execve()` の引数処理は少なくとも「CR3 切替前後で user pointer を裸でたどる」実装からは離れた。

### 11.5 user stack 確保方式の問題

- もともとの `alloc_user_stack()` は
  - `USER_STACK_PAGES - GUARD`
ページを一括で `pmm_alloc()` し、
  - それを連続物理メモリとしてそのまま user stack にマップ
する設計だった。
- `cpu1` child で `Exec: alloc_user_stack failed` が出たため、最初は stack サイズを `256` ページから `64` ページへ下げた。
- しかし 4 CPU 起動での再試験でも、`cpu1` child 側ではまだ `alloc_user_stack failed` が出た。
- この時点で、問題は stack size そのものより
  - 「連続物理メモリの一括確保」
が破綻していると判断した。

### 11.6 user stack を 1 ページずつ確保・マップする方式へ変更

- そこで `alloc_user_stack()` を全面的に切り替えた。
- 新方式では
  - 1 ページずつ `pmm_alloc(1)`
  - 1 ページずつ `vmm_map_page()`
  - `stack_write_bytes()` / `stack_write_u64()` で page 跨ぎ書き込み
という形にした。
- これに伴い `task_prepare_initial_user_stack()` も
  - 連続バッファ前提の `stack_mem + offset`
ではなく、
  - page 配列経由で stack を構築するように変更した。
- これで少なくとも「巨大な連続物理 stack 領域が取れずに失敗する」要因は取り除いた。

### 11.7 `get_current_task_raw()` の参照を `%gs:32` に変更

- `Exec: begin a=0 b=0` の不一致に対しては、`current_task` の参照方法を見直した。
- `syscall_entry.S` は cpu-local を `%gs:0` / `%gs:8` で直接見ている。
- 一方 `task.c` 側は `rdmsr(MSR_KERNEL_GS_BASE)` から `cpu_local*` を引いており、AP child の `execve()` ではこれが idle task を返していた。
- このため `get_current_task_raw()` は
  - `%gs:32`
から `current_task` を直接読むように変更した。
- なお `this_cpu()` 自体を `%gs:16` ベースへ切り替える案も試したが、これは early boot 側へ影響し、`Task system initialized.` 後の flow が壊れたため採用しなかった。
- 現在は
  - cpu-local 全体の取得は従来寄り
  - `current_task` だけを `%gs:32` から直読み
という折衷で置いている。

### 11.8 初回 user task 起動が shell 前で止まる新しい症状

- 上記の修正を積んだ段階で、新しく見えてきた症状がある。
- BSP の初回 user task 起動時に
  - `Starting first user task...`
  - `init: before first yield`
  - `kernel_yield: enter`
  - `schedule: cpu=0 cur=1`
  - `schedule: next=2 cpu=0 rsp=... top=0 ret=...`
までは出るが、その先の
  - `task_main: enter`
  - `Welcome to OrthOS Shell!`
は出ない。
- これは「初回の `schedule()` までは進んでいるが、`switch_context()` 後に `task_main` 入口へ到達していない」ことを示す。

### 11.9 `switch_context()` に渡している frame のエビデンス

- `schedule()` で next task の frame をログしたところ、
  - `next=2`
  - `cpu=0`
  - `rsp=18446603336792686528`
  - `top=0`
  - `ret=18446744071562099520`
を得た。
- ここで `ret` を `nm -n kernel.elf` で照合すると
  - `0xFFFFFFFF80007B40`
であり、これは `task_main` のアドレスと一致した。
- したがって、
  - 初回 task switch で選ばれる task は正しい
  - `ctx.rsp` 上の ret 先も `task_main`
であり、stack frame の大枠は壊れていない。
- このため、現在の本命は
  - `switch_context()` の `mov %cr3`
  - もしくはその直後の CPU 状態
である。

### 11.10 現時点の整理

- 今日の作業で進んだこと
  - AP child `execve` の不一致ログを取得
  - `fork_child_entry` の `swapgs` 問題を維持修正
  - `execve()` の user 引数コピー化
  - user stack の 1 ページずつ確保・マップ化
  - 初回 user task 起動停止点の詳細切り分け
- 逆に、まだ未解決なのは
  - 初回 `switch_context()` 後に `task_main` へ入れない理由
  - そのため `cpu1` child の `execve()` 再検証まで進め切れていないこと
である。

### 11.11 次回の予定

- 次回はここを続ける。
- 優先順位は以下の通り。
  1. `switch_context()` 後に `task_main` へ到達しない原因を解明する  
     候補: `cr3` 切替後の address space / FXSAVE-FXRSTOR / segment / GS 状態
  2. 初回 user task 起動を復旧する
  3. そのうえで再度 `/bin/testtime.elf` を 2 回実行し、`cpu1` child 側の `Exec: begin` と userspace 起動を再検証する
  4. もし初回起動復旧後も AP child で不一致が残るなら、`current_task` / cpu-local の取り扱いをさらに絞り込む

## 12. 初回 user task 停止の決着

### 12.1 まず `task_main` までは到達していた

- その後の切り分けで、
  - `schedule: next=2 cpu=0 ... ret=...`
  - `ABCtask_main: enter`
  - `D`
まで出ることを確認した。
- ここで
  - `A/B/C` は `switch_context()` の `cr3` 切替, `rsp` 切替, `popfq` 後
  - `D` は `call_app()` の `iretq` 直前
を意味する。
- したがって、
  - 初回 task switch は成立
  - `task_main()` にも入っている
  - さらに userspace への `iretq` 直前までも進んでいる
ことが実測できた。

### 12.2 `call_app()` の `swapgs` も疑った

- `fork_child_entry` で `swapgs` を外したのと同じ理屈で、`call_app()` 側の `swapgs` も疑った。
- 実際に `call_app()` から `swapgs` を外して再試験したが、見かけ上の症状はそれだけでは解消しなかった。
- ただし後述のとおり、`iretq` 周辺の fault 経路で `GS` が絡んで二次故障していたため、この修正自体は維持する方が自然である。

### 12.3 `crt0_musl` 冒頭に `ud2` を置いて userspace 遷移そのものを検査

- userspace に本当に入っているかを切り分けるため、`user/crt0_musl.S` の `_start` 冒頭へ一時的に `ud2` を置いた。
- しかし当初は画面上で `#UD` が見えず、見かけ上は
  - `D`
の直後に無言停止するように見えた。
- この段階では「`iretq` 自体が失敗しているのか」「userspace fault が別の故障で覆い隠されているのか」が未確定だった。

### 12.4 `-d int` で本当に起きていた fault を確認

- そこで QEMU を
  - `-d int -D /tmp/orthox-qemu.log`
で起動し、割り込み・例外ログを直接取得した。
- 最初に見えたのは、
  - `CR2=0x20`
  - `RIP=0xffffffff80005d24`
  - 無限 `#PF`
であった。
- `0xffffffff80005d24` を `objdump` で引くと、これは `get_current_task()` の
  - `mov %gs:0x20, %rax`
に一致した。
- つまり、userspace 遷移前後に何らかの例外が起きた際、kernel 側が `%gs:32` 直読みで `current_task` を引こうとして `GS=0` のまま二次故障し、本来の最初の例外が隠れていたことが分かった。

### 12.5 `%gs:32` 直読みは切り分け用としては有効だったが、本修正としては不適切だった

- `get_current_task_raw()` を `%gs:32` 直読みへ変えたのは、
  - `[smp] fork pid=4 cpu=1`
  - `Exec: begin a=0 b=0`
という AP child の不一致を観測するためには有効だった。
- しかしその方式は、
  - `iretq` 周辺
  - 例外遷移の途中
で `GS` が安定していない瞬間に極めて弱い。
- その結果、初回 user task の本来の例外を見えなくしていた。
- そこで `get_current_task_raw()` は
  - `this_cpu()->current_task`
へ戻した。

### 12.6 `get_current_task_raw()` を戻したことで本来の `#UD` が表に出た

- この変更後、再度 `crt0_musl` 冒頭に `ud2` を置いて起動したところ、今度は画面上で素直に
  - `*** EXCEPTION OCCURRED ***`
  - `Vector: 6`
  - `RIP: 0x0000000000400003`
が出た。
- これは
  - `iretq` 自体は成功
  - userspace `_start` には到達
  - `ud2` が意図どおり `#UD` を起こした
ことの直接エビデンスである。
- つまり、初回 user task 停止の本質は「userspace に入れていない」ことではなく、「その前後の kernel fault handling が `%gs:32` 直読みで二次故障していた」ことだった。

### 12.7 `ud2` を外すと shell は復旧した

- 最後に `crt0_musl.S` から `ud2` を外し、`_start` を通常状態へ戻した。
- その状態で `-smp 4` 起動すると、
  - `Starting first user task...`
  - `Welcome to OrthOS Shell!`
  - `/$`
まで正常に進んだ。
- さらに serial から `pwd` を送ると
  - `/`
が返ってくることも確認した。

### 12.8 今回の最終整理

- 時間を使った本当の問題は、
  - AP child `execve` の切り分けのために入れた `%gs:32` 直読みが、
  - 初回 user task の fault path を壊し、
  - 本来の例外を二次故障で覆い隠していた
ことだった。
- そのため今回の最終状態では、以下を本修正として残している。
  1. `get_current_task_raw()` は `this_cpu()->current_task` を使う
  2. `call_app()` からは `swapgs` を外す
  3. `task_main()` は `argc/argv` をそのまま `call_app()` に渡す
- 逆に、
  - `A/B/C/D`
  - `crt0` の `U`
  - `ud2`
  - `schedule` / `kernel_yield` の詳細ログ
など、切り分け専用の一時デバッグはすべて除去した。

### 12.9 次回の作業再開点

- 初回 shell 復旧までは確認できたので、次回は本来の SMP 本筋へ戻る。
- 優先順位は以下の通り。
  1. AP child `/bin/testtime.elf` を再度 2 回実行し、`cpu1` child 側の `fork/execve` を再検証する
  2. まだ残っている `cpu0` 固定前提を洗い出す
  3. `BKL` 前提のまま、user task を他 CPU へ安全に広げる準備を続ける

### 12.10 このステップの意味

- 今回の到達点は、単なる
  - AP bring-up
  - idle loop
  - local timer / IPI 確認
の延長ではない。
- 実際には、
  - 初回 user task の `iretq`
  - userspace `_start`
  - shell 起動
  - serial 入力
までを `-smp 4` のまま復旧した。
- これは `Orthox-64` の SMP 作業が
  - 「複数 CPU が起動するか」
という bring-up 段階から、
  - 「userland を含む本番経路をどう多核で成立させるか」
という段階へ進んだことを意味する。
- 特に今回得た知見で重要なのは、
  - userspace に入れていないのではなく、
  - 例外経路の `current_task` 参照が `%gs:32` 直読みで二次故障していた
とエビデンスベースで切り分けられた点である。
- この種の問題は、kernel-only の SMP 実験では見えず、
  - `fork`
  - `execve`
  - `iretq`
  - userspace fault
まで実際に通そうとして初めて露出する。
- したがって、今回の復旧は単なるバグ修正ではなく、
  - `Orthox-64` が userland を含む SMP の実運用問題に入った
ことを示す、非常に重要なステップだった。

## 13. AP child `execve` の再検証と userspace 実証

### 13.1 再開時点の症状

- 前回までで、`-smp 4` でも初回 shell 起動自体は復旧していた。
- ただし SMP 本筋の確認として、`/bin/testtime.elf` を 2 回実行すると、
  - 1 回目は `cpu0` 側 child として成功
  - 2 回目は AP 側 child を引いた直後に失敗
という状態だった。
- 当初のログは
  - `Exec: alloc_user_stack failed`
  - 続いて userspace 側 `#UD`
であり、見かけ上は「user stack 確保失敗」に見えていた。

### 13.2 まず reaping と address space 解放を整理した

- 先に、単なるリークの可能性を潰した。
- `wait4()` で zombie child を論理的に消すだけではなく、`task_reap()` を導入して
  - `task_list` から unlink
  - child の old `cr3` を `vmm_free_user_pml4()` で解放
  - child kernel stack を解放
  - `task` 構造体自体を解放
する形に直した。
- 加えて `execve` 成功時にも、fork 直後に child が持っていた旧 user page table を解放するようにした。
- これは設計上必要な修正であり、そのまま本修正として残す。

### 13.3 `alloc_user_stack failed` は誤誘導だった

- その後、`alloc_user_stack()` と `task_execve()` の境界に段階ログを入れて再度観測した。
- その結果、
  - `Exec: before prepare stack`
  - の直後に
  - `Exec: alloc_user_stack failed`
が出る一方、
  - `alloc_user_stack enter`
  - `alloc_user_stack ok`
は出ないケースがあった。
- つまり、最初に疑っていた
  - 物理メモリ枯渇
  - page 0 を `NULL` 扱いしているだけ
だけでは説明できず、
  - `task_prepare_initial_user_stack()` に渡る時点の `current task` または cpu-local 参照
が壊れている可能性が高まった。

### 13.4 `this_cpu()` の参照基準が AP syscall 経路で壊れていた

- 根因は `this_cpu()` 側にあった。
- それまで `this_cpu()` は `MSR_KERNEL_GS_BASE` を読んでいたが、syscall 入口では `swapgs` 後に live な cpu-local は `GS_BASE` 側へ出てくる。
- BSP 単独では問題化しにくかったが、AP child の `execve` ではこの差が露出し、
  - `get_current_task_raw()` が AP child ではなく別 task を見たり、
  - その結果として stack 構築前提が崩れる
という形で現れていた。
- 修正として、
  - `this_cpu()` はまず live な `MSR_GS_BASE` を読む
  - fallback として `MSR_KERNEL_GS_BASE` も見る
形へ変更した。
- これにより AP child 側でも `task_execve()` が正しい `current_task` を見られるようになった。

### 13.5 それでも残った `#UD` の正体

- `this_cpu()` 修正後、AP child の `/bin/testtime.elf` は
  - `Exec: before prepare stack`
  - `alloc_user_stack enter`
  - `alloc_user_stack ok`
  - `Exec: prepare stack ok`
まで進むようになった。
- つまり `execve` と user stack 構築自体は通過していた。
- しかしその直後に
  - `#UD @ 0x0000000000402167`
で停止した。
- `objdump` で確認すると、`0x402167` は
  - `xorps %xmm0, %xmm0`
であり、userspace 側の SSE 命令だった。
- ここから、AP child は
  - すでに user mode へ入っている
  - ただし AP コアでは SSE/FPU が有効化されていない
と判明した。

### 13.6 AP 側でも SSE と paging feature を有効化した

- これまで `enable_sse()` と `enable_paging_features()` は BSP の early bring-up でしか呼ばれていなかった。
- そのため AP は
  - local timer
  - syscall MSR
  - GDT/IDT/TSS
は持っていても、
  - CR0/CR4 の SSE 関連ビット
は未設定だった。
- `kernel/smp.c` の AP entry に
  - `smp_enable_sse()`
  - `smp_enable_paging_features()`
を追加して、AP も BSP と同じ前提で userspace を走らせるようにした。
- この修正後、2 回目の `/bin/testtime.elf` も正常完走するようになった。

### 13.7 実証用に `get_cpu_id()` と `showcpu.elf` を追加した

- ここまでで「壊れなくなった」ことは確認できたが、
  - 2 回目が本当に AP で走っているか
を userspace から直接観測したかった。
- そのため最小の private syscall として
  - `ORTH_SYS_GET_CPU_ID`
を追加し、kernel では `get_cpu_local()->cpu_id` を返すようにした。
- userspace 側 wrapper も `syscalls.c` / `syscalls_musl.c` に追加した。
- さらに検証用の小さな program `showcpu.elf` を追加し、
  - `cpu=%d ticks=%llu`
を出すようにした。

### 13.8 最終確認結果

- `-smp 4` で起動し、shell 到達を確認したあと、
  - `/bin/showcpu.elf`
  - `/bin/showcpu.elf`
を 2 回実行した。
- 結果は
  - 1 回目: `cpu=0`
  - 2 回目: `cpu=1`
だった。
- これは
  - 2 回目の child が実際に BSP ではない CPU 上で userspace 実行されている
ことの直接エビデンスである。
- 同様に `/bin/testtime.elf` を 2 回連続実行しても、両方とも正常完走した。

### 13.9 今回残した本修正

- 今回の作業で最終的に残すべき修正は次の通り。
  1. `task_reap()` と `vmm_free_user_pml4()` による child 資源解放
  2. `execve` 成功時の旧 user page table 解放
  3. PMM で physical page 0 を常時予約
  4. `this_cpu()` を live `GS_BASE` ベースに修正
  5. AP entry で SSE / paging feature を有効化
  6. `ORTH_SYS_GET_CPU_ID` と `showcpu.elf` による userspace 実証

### 13.10 この到達点の意味

- 今回の到達点は、
  - AP が起動する
  - AP が idle loop に入る
という bring-up の延長ではない。
- 実際には
  - AP child が `fork`
  - `execve`
  - userspace entry
  - musl program 実行
まで完了し、
  - `cpu=1`
を userspace から返せるところまで来ている。
- つまり `Orthox-64` の SMP は、
  - 「AP が生きている」
段階から、
  - 「AP 上で userland を現実に走らせられる」
段階へ進んだ。
- これは SMP 全体の中でも非常に大きなマイルストーンであり、次の per-CPU scheduling や `cpu0` 固定解除に進むための重要な前提が成立したと言える。

## 14. 明日の作業予定

- 今日の時点で、`Orthox-64` は AP 上で userspace program を実行し、`showcpu.elf` で `cpu=1` を返せるところまで来た。
- したがって次の優先順位は、単なる bring-up ではなく「実用的な multi-CPU scheduling へ寄せること」になる。

### 14.1 優先順位 1: `cpu0` 固定前提の棚卸し

- まず、まだ暗黙に `cpu0` を前提にしている経路を洗い出す。
- 特に確認対象は以下。
  1. `task_create` / `task_fork` / wakeup 側での affinity 付与
  2. timer tick 由来の wakeup と resched
  3. network / console / wait 系の wakeup がどの CPU を起こす想定か
- ここを整理しないまま task migration に進むと、再び BSP 偏重の動作に戻る可能性が高い。

### 14.2 優先順位 2: per-CPU scheduler への橋渡し

- 現状でも AP 上で userland は走るが、scheduler はまだ本格的な per-CPU run queue ではない。
- 次は
  - affinity を明示的に変えた task がどの CPU で拾われるか
  - wakeup 後に resched IPI が正しく効いているか
を、小さな実験 program で増やしていく。
- 可能なら `showcpu` を少し拡張して、
  - `fork` 後の親子で CPU を表示
  - `sleep` 後の再開 CPU も表示
といった観測を追加したい。

### 14.3 優先順位 3: BKL 前提の安全域を広げる

- 現状は coarse `BKL` のうえで SMP を成立させている段階であり、これは正しい。
- ただし今後 task が他 CPU へ広がると、
  - syscall 中の wait/wake
  - fd/socket/file の共有状態
  - signal 配送
あたりで、BKL 前提が崩れていないかを再点検する必要がある。
- 明日はまず「危ないところを増やさない」ための棚卸しを優先し、いきなり lock 分解には進まない。

### 14.4 現時点での大局整理

- 今日の成果で、SMP はおおよそ `65%` 前後まで来たと見てよい。
- ただし残っている本丸は
  1. `cpu0` 固定の削減
  2. 実質的な multi-CPU scheduling
  3. その先の `pthread/futex`
である。
- したがって明日の作業は、
  - 新機能追加より
  - scheduler の前提整理と観測強化
に寄せるのが正しい。

## 15. 追記: per-CPU run queue 後の安定化と次の壁

- `d3727d1` で per-CPU run queue の初期版を入れたあと、次の step として
  - `fork` child の CPU 配置を queue の混み具合で選ぶ
  - remote CPU の ready task が idle AP に埋もれないようにする
を試した。

### 15.1 load-aware fork 配置

- まず `task_fork()` の child 配置を、単純な round-robin ではなく
  - `runq_count`
  - その CPU で現在動いている non-idle task
を足し合わせた軽い load 指標で選ぶようにした。
- 実装は `kernel/task.c` の `task_cpu_load_locked()` と `choose_spawn_cpu_locked()` で、
  - fallback CPU を親 affinity
  - tie break は従来の `next_cpu`
とした。
- これ自体は build を壊さず、`forkcputest` の child 分散も自然に見えた。

### 15.2 本当に効いた修正: remote READY 後の resched IPI

- その後の実機試験で、`smpstress` は
  - `pipestress` は通る
  - しかしその後の `forkcputest` / `reaptest` に進む前後で hang する
という形で止まった。
- 原因の切り分けから見えたのは、
  - `task_wake()` は remote CPU に対して `task_request_resched_cpu()` を打つ
  - しかし `task_mark_ready_on_cpu()` と `task_fork()` の新規 child 投入は ready queue に入れるだけ
という非対称性だった。
- そのため、idle AP 側では
  - ready task は存在する
  - しかし resched 契機が弱く、すぐ拾われない
という窓が残っていた。
- そこで `kernel/task.c` で以下を入れた。
  1. `task_mark_ready_on_cpu()` で enqueue 成功後に target CPU へ `task_request_resched_cpu()`
  2. `task_fork()` で child enqueue 後に `spawn_cpu` へ `task_request_resched_cpu()`
- この変更後、再現していた `smpstress` の先頭 hang は消えた。
- SMP 4CPU の default spread で
  - `/bin/smpstress.elf 2`
  - `/bin/smpstress.elf 6`
が再び通るところまで戻せた。

### 15.3 試したが戻したもの: 粗い work stealing

- 次の step として、idle CPU が他 CPU の run queue 末尾から task を steal する簡易版も試した。
- ただしこれは早い段階で regression を出した。
  - `smpstress` 中の `reaptest` で CPU2 が exception に落ちた
  - つまり task migration の安全条件がまだ足りていない
- このため、その実験はその場で revert した。
- 現時点で残してよいのは
  - load-aware fork 配置
  - remote READY 後の resched IPI
までであり、
  - 汎用 task stealing
はまだ入れる段階ではないと判断した。

### 15.4 この時点の正しい次手

- per-CPU run queue 化の次に必要なのは、
  - 「どの task が他 CPU へ移してよいか」
  - 「wait 中 / syscall 中 / parent-child 連携中に移してはいけない task は何か」
を明確にすることだ。
- つまり次の本命は、雑な stealing ではなく
  - migration eligibility の定義
  - safe point の設計
  - それを見られる診断の追加
である。
- 今回の段階で無理に load balancing を進めるより、
  - remote enqueue の確実な起床
  - per-CPU run queue の安定運用
を先に固める方が正しい。
