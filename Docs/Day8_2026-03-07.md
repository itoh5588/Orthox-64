# Day 8 作業ログ (2026-03-07)

## 本日の実装内容と達成事項

本日は、orthOS における「Cコンパイラのセルフホスト（OS上でのCコードのコンパイル）」という巨大なマイルストーンを達成しました。

### 1. プロセス間通信（パイプとリダイレクト）の実装
GCC のようなコンパイラドライバが `cc1` → `as` → `ld` と複数のプロセスを連携させるために不可欠なシステムコールを実装しました。
- `sys_pipe` (22) と `sys_dup2` (33) をカーネルに実装。
- `FT_PIPE` ファイルタイプを新設し、リングバッファを用いたプロセス間のデータ転送とブロック（`schedule()` による待機）を実現。
- `fork` 時にファイル記述子（FD）が子プロセスに継承されないバグを修正。
- **バグ修正**: パイプ構造体 (`pipe_t`) が物理ページ（4096バイト）をはみ出し、子プロセスのスタックを破壊して General Protection Fault を引き起こす致命的な問題を、バッファサイズ (`PIPE_BUF_SIZE`) を 4000 に縮小することで解決。

### 2. GCC (cc1) のクロスビルドとOSスタブの拡充
当初 GCC 4.9.4 の移植を試みましたが、ホストOS（macOS）のクロスコンパイラが C++ 標準ライブラリを持たない純粋なベアメタル環境であったため、GCC 自身のビルド要件（C++化）を満たせず頓挫しました。
これを解決するため、**「完全に C言語のみで書かれている最後の GCC」であるバージョン 4.7.4** にターゲットを切り替えました。
- `user_stubs.c` に `mkdir`, `sleep`, `alarm`, `execv`, `chdir`, `gettimeofday`, `mmap`, `munmap` などの POSIX 互換ダミー関数を追加し、`libosstubs.a` として GCC ツール群にリンク。
- GCC が大量のメモリを確保する際に使用する `mmap` を、ページアラインされた `malloc` (`memalign`) を用いて代用するよう実装。
- macOS の `awk` のソート非互換性によるビルドエラーを、GNU awk (`gawk`) と `LC_ALL=C` を指定することで回避。

### 3. orthOS 上での C言語セルフコンパイルと実行の成功
構築した `cc1.elf` を OS 上のファイルシステム (`rootfs.tar`) に配置し、Cプログラムのコンパイルテストを行いました。
- `cc1.elf` が正しくメモリを確保し、構文解析と最適化を経てアセンブリファイル (`test.s`) を生成。
- 自作コンパイラドライバ (`user/gcc.c`) が `cc1` → `as` → `ld` のパイプラインを自動実行。
- 最終的に生成された `a.out` が `10 + 20` を計算するプログラムを実行し、OS に終了コード `30` (ウェイトステータス `0x1E00`) を返したことを確認。
これにより、**OS 上での C言語のコンパイルから実行までのフルサイクルが完全に動作する** ことが証明されました。

## 今後の課題・次へのステップ
- **ELFローダの改善**: 実行ファイル内の文字列定数（`.rodata`）が正しく出力されなかったため、ELFのセグメントロード処理やリンカスクリプトの調整が必要。
- **ファイルシステムの拡充**: 現状の `mmap` は単なる `malloc` のラッパーであり、本当のファイルバックト・マッピングや動的ページングは未実装。 FAT32 などの本格的なファイルシステムの導入と共に仮想メモリ管理を強化する。
- **セルフホストに向けた make などの移植**: 次のステップとして、プロジェクト全体を OS 内部でビルドできるように、ビルド自動化ツール (`make`) などを移植する。

## 明日の予定 (Day 9)

1. **プリエンプティブ・マルチタスクの実現**
   - 現在のタスク切り替えはシステムコール呼び出し時などに依存する「協調的マルチタスク」です。
   - タイマー割り込み（Vector 32 / LAPICタイマー）ハンドラから強制的に `schedule()` を呼び出し、プロセスを時分割で切り替える「プリエンプティブ・マルチタスク」を実装します。これにより、無限ループするプログラムがあってもシステム全体がフリーズしなくなります。

2. **Copy-On-Write (COW) の実装と fork() の高速化**
   - 現在の `fork()` は、親プロセスのメモリ空間（ページテーブル）を物理的にすべてコピーしているため、巨大なメモリを消費する GCC (`cc1`) などのプロセスで `fork` を多用すると非常に非効率です（あるいはメモリ枯渇の原因になります）。
   - `fork` 時にはページテーブルの権限を「読み取り専用」として共有し、どちらかのプロセスが書き込みを行った際に発生するページフォルト (Page Fault) をトリガーにして初めて物理メモリをコピーする「コピーオンライト (Copy-On-Write)」の仕組みを `kernel/vmm.c` に実装します。

3. **不足しているシステムコールの拡充と ELFローダの改善**
   - 文字列定数 (`.rodata`) が正しく出力されなかった問題（ELF セグメントのロード処理バグの可能性）を修正します。
   - `mmap` を単なる `malloc` ではなく、真の「ファイル内容のメモリマッピング（または無名ページの遅延割り当て）」として正しく機能するように VFS や VMM と連携させます。
   - gcc 以外のツール（`make` や `bash` 等）の移植に向けて、不足しているシステムコール（`sigaction`, `ioctl` など）の調査とスタブ拡充を行います。

4. **USB機能の対応 (xHCI / マスストレージなど)**
   - ファイルシステムやデバイスの利用の幅を広げるため、USBホストコントローラ (xHCI) ドライバおよび USB マスストレージクラスへの対応を検討・実装します。これにより実機でのUSBメモリ起動や読み書きを目指します。

## 【付録 1】 現在の Newlib (C標準ライブラリ) の移植ステータス

現在、orthOS の中での Newlib（C標準ライブラリ）は、「スタティックリンクライブラリとしてシステムコールと結合され、完全に動作する状態」になっています。

- **ライブラリの配置**: 
  - `user/include/` に `stdio.h`, `stdlib.h`, `math.h` などの標準ヘッダを完備。
  - `user/libs/libc.a` および `libm.a` としてクロスコンパイル済みの静的ライブラリを配置。
- **システムコール・スタブ (`user/syscalls.c`)**: 
  - Newlib が内部で呼び出す低レイヤ関数群（`_write`, `_read`, `_open`, `_close`, `_sbrk`, `_stat`, `_fstat`, `_lseek`, `_pipe`, `_dup2` など）をすべて orthOS の `syscall` 命令へルーティング。
- **実績**: 
  - Day 2 の段階で `printf` や `malloc` が動作。
  - Day 8 にて、この Newlib 上で巨大なアプリケーションである GCC (`cc1`) がメモリ管理 (`sbrk`, `mmap` 代替) やファイルI/Oを正常に行い、クラッシュすることなく動作したことで、基盤としての堅牢性が完全に証明された。

## 【付録 2】 OS自作本 出版に向けた章立て構想 (KDP 等での出版案)

本プロジェクトの軌跡は、「フルスクラッチのOS上で本物のGCCを動かす（セルフホスト）」という、既存の入門書にはない圧倒的なクライマックスを持っています。内容の濃さから**「上下巻（2分冊）」**での構成が最適と考えられます。

### 上巻：大地を創る（ブートローダからマルチタスクまで）
「ゼロからOSの土台を作り、複数のプログラムが同時に動く世界を構築する」
*   **第1章：未知への旅立ち** - Limineブートローダの恩恵と、64ビット環境の初期化。
*   **第2章：メモリという広大な土地** - 物理メモリ管理 (PMM) と ページング (VMM) による仮想空間の構築。
*   **第3章：神の視点と割り込み** - IDTの設定、例外ハンドリング、LAPICタイマー。
*   **第4章：ユーザー空間への降下** - GDT/TSSの再構築、リング3への移行と Triple Fault の罠。
*   **第5章：分身するプロセス** - システムコールの実装、`fork`と`execve`、そして協調的マルチタスク。

### 下巻：文明の夜明け（C標準ライブラリからGCCセルフホストまで）
「OSと外界（既存ソフトウェア）を繋ぎ、自立した開発環境（セルフホスト）を手に入れる」
*   **第6章：外界との架け橋** - VFS（仮想ファイルシステム）の設計と Ramfs/TAR ファイルシステムの統合。
*   **第7章：C標準ライブラリの移植** - Newlib の導入と POSIX スタブ関数の実装 (`_write`, `_sbrk` など)。
*   **第8章：プロセス間通信** - リングバッファによる `pipe` と `dup2` の実装。コンパイラドライバを動かす準備。
*   **第9章：巨人の移植（GCCクロスビルドの死闘）** - Binutils と GCC の移植。C++依存の罠、`mmap`とガベージコレクタの衝突、そして `AWK` のバグを越えて。
*   **第10章：セルフホストの達成** - 自作OS上で「10+20」がコンパイルされた瞬間。今後の展望（プリエンプションやUSB対応など）。

## 【付録 3】 真の究極目標：「DOOM」を動かす

GCCのセルフホストを達成した orthOS-64 の次なる（そして究極の）目標は、**「自作OSの上で伝説のゲーム『DOOM』をコンパイルし、動かすこと (It runs DOOM!)」** です。
現在のOSには、DOOMの移植（`doomgeneric` 等を利用）に必要ないくつかのピースが欠けており、今後はこれを埋めることをマイルストーンに加えます。

### DOOM 移植に向けた3つの実装課題
1. **フレームバッファの描画 (`DG_DrawFrame`)**
   - Limineブートローダが用意した高解像度のピクセルメモリ（フレームバッファ）のアドレスをカーネルから取得し、ユーザープログラムがそこに直接色データを書き込めるシステムコール（`mmap` 等による共有）を実装します。
2. **リアルタイムなキーボード入力 (`DG_GetKey`)**
   - ゲームの移動や射撃に必要な「キーの押下・解放」をリアルタイムに検知し、ユーザー空間のDOOMエンジンへ非同期で伝えるためのイベントキューとドライバを実装します。
3. **ミリ秒精度のタイマー (`DG_GetTicksMs`, `DG_SleepMs`)**
   - LAPICタイマー割り込みを利用してシステムの経過時間（Tick）を計測し、OSスタブの `gettimeofday` などを通じて正確なミリ秒を取得・待機できるようにします。
4. **サウンド再生 (Audio Support)**
   - 「音のないDOOMはDOOMではない」という理念のもと、ゲーム体験を完成させるための音声出力を目指します。まずは I/Oポート制御による PCスピーカー (BEEP音) でのレトロなBGM再生から始め、将来的には PCIバス経由での Intel HDA (High Definition Audio) または AC97 ドライバの実装に挑戦し、迫力ある効果音と音楽の再生を実現します。

OS本としても、「自作OSでコンパイラを動かし、そのコンパイラでゲームをビルドして遊び、音まで鳴らす」というこれ以上ない最高のクライマックスになる予定です。
