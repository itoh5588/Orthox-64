# Day14 (2026-03-14)

## musl について

`musl` は POSIX/Unix 互換を強く意識した軽量 libc であり、OrthOS が目指している
「POSIX 互換で既存 OSS を多く取り込める OS」という方向と相性が良い。

`newlib` は最小 libc として足場作りには有効だったが、互換不足を OrthOS 側で厚く補う必要があった。
一方で `musl` は libc 自体が持つ互換性が高く、OS 側は syscall / VFS / signal / tty の筋を通すことに集中しやすい。

そのため OrthOS では、

1. 既定系は `newlib` のまま維持
2. `musl` は並行経路で評価
3. 差分を観測しながら段階的に移行

という方針を取っている。

## 今日の技術的修正

`newlib` 用の `user/syscalls.c` は互換補完が厚すぎるため、`musl` と直接ぶつかる。

## 今日わかったこと

### 1. musl への完全切替はまだ不要

今必要なのは一括移行ではなく、`newlib` と `musl` を横で比較しながら、どこに POSIX 互換の不足があるかを可視化すること。

### 2. musl は OrthOS の目標に近い

OrthOS の目標は POSIX 互換で既存 OSS を多く取り込めることにある。
その意味で `musl` は `newlib` より方向性が合っている。

### 3. 既存系を壊さずに進める方が正しい

既定 shell や USB root 系が今動いている以上、libc 移行は並行経路で育てるべきである。

### 4. musl 全般はもう起動している

`/bin-musl/usertest` は正常に起動した。したがって、musl の基本 ABI や `execve` の最低限は通っている。

### 5. BusyBox musl の初期クラッシュは解消済み

当初は `/bin-musl/ash` が以下で落ちていた。

- `#PF(User): read/page-not-present at 0x0000000000000000`
- `RIP: 0x0000000000401CB0`

その後の `execve` stack, `crt0`, TLS, `arch_prctl`, `getdents64`, `ioctl/writev/exit_group` 周辺の修正により、この問題は解消した。

## さらに追記: `ls` / `readdir` の追跡と修正

- musl `opendir()` は Linux の `O_DIRECTORY=0x10000` を使っていたが、kernel 側は旧値 `0x200000` を見ていた
- そのため `opendir("/bin")` が通常 file open として扱われ、`readdir()` 側で空振りしていた
- kernel では `O_DIRECTORY` を Linux 値へ合わせつつ、旧値 `0x200000` も受理する互換処理を追加した
- あわせて active root のサブディレクトリ列挙を修正し、ramfs 側の空ディレクトリが tar 側を覆い隠さないようにした
- `tar_split_child()` には `bin` と `bin-musl` を誤マッチしない prefix 境界チェックを追加した

さらに、BusyBox `ls` は raw syscall 経路で以下を要求していたため、kernel に最小実装を追加した。

- `ioctl(16)`
- `writev(20)`
- `set_tid_address(218)`
- `exit_group(231)`

`ioctl` は少なくとも以下を扱う。

- `TIOCGWINSZ`
- `TCGETS`
- `TCSETS`
- `TIOCGPGRP`
- `TIOCSPGRP`

QEMU 確認では、以下を再現・修正できた。

- `/bin/readdirtest /bin` で `cat`, `echo`, `ls`, `ash` などの列挙が返る
- `/bin/ls /bin` で applet 一覧が表示される

この時点で、musl userland の既知の `ls` 無言問題は解消した。

## さらに追記: syscall 番号の整理

`getdents64(217)` 問題を起点に、`include/syscall.h` の番号定義を見直した。

整理方針:

- Linux x86_64 に存在する syscall は Linux の実番号へ合わせる
- OrthOS 専用 syscall は `ORTH_SYS_*` へ改名し、`1000+` の独自帯へ移す

主な補正:

- `SYS_KILL` を `62` へ修正
- `SYS_MKDIR` を `83` へ修正
- `SYS_RMDIR` を `84` へ修正
- `SYS_SETPGID` を `109` へ修正
- `SYS_GETPGRP` を `111` へ修正
- `SYS_SETSID` を `112` へ修正
- `SYS_GETDENTS` を Linux 互換の `78` として定義

独自帯へ移したもの:

- `ORTH_SYS_LS`
- `ORTH_SYS_GET_VIDEO_INFO`
- `ORTH_SYS_MAP_FRAMEBUFFER`
- `ORTH_SYS_GET_TICKS_MS`
- `ORTH_SYS_SLEEP_MS`
- `ORTH_SYS_GET_KEY_EVENT`
- `ORTH_SYS_SOUND_*`
- `ORTH_SYS_USB_*`
- `ORTH_SYS_MOUNT_*`
- `ORTH_SYS_GET_MOUNT_STATUS`
- `ORTH_SYS_TC*`
- `ORTH_SYS_SIG*`

これにより、musl や既存 OSS が raw Linux syscall 番号を直接使ったときに、OrthOS 独自番号と衝突しにくい状態になった。

確認:

- `make orthos.iso` 成功
- QEMU 上で `/bin/ls /bin` 成功

## さらに追記: FS metadata の共通化に着手

- `kstat` を `mode,size` だけでなく、`uid`, `gid`, `nlink`, `atime`, `mtime`, `ctime` を持つ形へ拡張した
- `st_dev` と `st_ino` も返すようにし、backend ごとに device 番号を分け、inode は path/hash ベースで安定化した
- ramfs は作成時に `0644` / `0755` と時刻を持ち、`read` で `atime`、`write` / `chmod` / `rename` で `mtime` / `ctime` が更新されるようにした
- tar root は header の `mode`, `uid`, `gid`, `mtime` を読み出して `stat` に反映するようにした
- `fstat` は open 済み fd の backend を見て `stat(path)` と整合する metadata を返すように整理した
- この変更に合わせて newlib / musl の syscall wrapper も `struct stat` へ `uid`, `gid`, `nlink`, `timespec`, `blocks`, `blksize` を詰めるようにした

確認:

- `make busybox-ash-musl-install orthos.iso` 成功
- `bash tests/musl_smoke.sh` 成功

## さらに追記: musl Doom の現状課題

`doomgeneric` の musl ビルドと WAD 読み込みは通ったが、当日確認時点では性能と音に未解決課題が残っている。

- musl 版 Doom は起動し、`doom1.wad` の認識、`DG_Init`、`I_InitGraphics` までは到達する
- ただしタイトル画面が出るまで体感で 20 秒前後かかる
- 初期化ログの実測では、主なボトルネックは `R_Init` で約 13 秒
  - `R_InitTextures`: 約 3.3 秒
  - `R_InitSpriteLumps`: 約 9.7 秒
- `HU_Init` と `ST_Init` も追加でそれぞれ約 1.3 秒、約 1.6 秒かかっている

音については、SB16 自体は無効ではない。

- QEMU の musl Doom 実行ログでは `[sb16] initialized` が出ており、DSP reset handshake は成功している
- それでも現時点ではゲーム中の効果音が期待通りに聞こえない
- したがって、問題は「SB16 が無い」ではなく、Doom 側の再生経路または実行フローの不整合として追う必要がある

当日中に行った追加切り分け:

- `readv(19)` 未実装により musl `stdio` が `fread=0` になっていた問題は kernel 側の `SYS_READV` 実装追加で解消
- `doomgeneric` の musl ビルドは `FORCE` 依存を付け、古い `doom-musl.elf` を掴み続けないよう修正
- `R_InitSpriteLumps` は各 sprite lump を丸ごと `W_CacheLumpNum()` せず、ヘッダのみ `W_Read()` で直読むように変更
- `doomgeneric` のビルド最適化は `-Os` から `-O2` に変更

残件:

- musl Doom の初期化性能改善
  - 特に `R_InitTextures` と `R_InitSpriteLumps` の時間短縮
- SB16 初期化後に実際の効果音が出ない理由の切り分け
- shell 経由起動時に混ざる入力や終了挙動の整理

## さらに追記: `gcc/as/ld` 復旧の途中経過

musl userland と Doom の整理の次に、newlib 系の toolchain (`gcc.elf`, `cc1`, `as`, `ld`) を current kernel / libc ABI に追随させる作業を行った。

まず、Day12 時点の `as-new` / `ld-new` / `cc1` は 2026-03-11 にビルドされたままの成果物で、当日の `kstat` ABI 拡張に追随していなかった。

- 旧 `struct kstat` は
  - `mode`
  - `size`
  のみを持つ 16-byte 相当のレイアウトだった
- 現在の kernel は `dev`, `ino`, `uid`, `gid`, `nlink`, `rdev`, `atime`, `mtime`, `ctime` を含む大きい `kstat` を `SYS_FSTAT` / `SYS_STAT` で返す
- その結果、古い `syscalls.o` を抱えた `as` / `ld` / `cc1` は `fstat` で小さい stack buffer を渡し、stdio 初期化中に stack を壊していた

このため、以下を実施した。

- `Makefile` に toolchain 再リンク経路を追加
  - `user/crt0.o`
  - `user/syscalls.o`
  - `ports/user_stubs.o`
  - `ports/libosstubs.a`
  - `ports/gcc-4.7.4/build/gcc/cc1`
  - `ports/binutils-2.26/binutils-2.26/build/gas/as-new`
  - `ports/binutils-2.26/binutils-2.26/build/ld/ld-new`
- `ports/libosstubs.a` は `user_stubs.o` のみを含むように整理し、`crt0.o` / `syscalls.o` の重複混入を解消
- `ports/user_stubs.c` の stub 群は weak 定義に変更し、current `syscalls.o` と衝突しないようにした
- `user/syscalls.c` の `getopt` / `optarg` / `optind` / `opterr` / `optopt` / `optreset` も weak 化し、newlib 本体の `getopt` と共存できるようにした

さらに、newlib 側 `crt0` で stack alignment が崩れていた。

- `as --version` の current build は `findfp.c` の `global_stdio_init.part.0` で `movaps` により `#GP` を起こしていた
- 原因は `user/crt0.S` が `main` 呼び出し前に SysV x86_64 ABI の 16-byte alignment を満たしていなかったこと
- `user/crt0.S` に `andq $-16, %rsp` を追加して修正

あわせて、shell と `gcc.elf` が stale binary を掴まないよう配置も揃えた。

- `rootfs/bin/as`
- `rootfs/bin/ld`
- `rootfs/bin/cc1.elf`
- `rootfs/bin/gcc.elf`
  を `rootfs.tar` 作成前に current build から毎回同期するようにした
- `user/gcc.c` は `/bin/cc1.elf`, `/bin/as`, `/bin/ld` を明示的に呼ぶよう変更した

確認結果:

- `make toolchain` 成功
- `make orthos.iso` 成功
- QEMU 上で `as --version` 成功
  - `GNU assembler (GNU Binutils) 2.26.20160125`
- QEMU 上で `ld --version` 成功
  - `GNU ld (GNU Binutils) 2.26.20160125`

この時点で、`as` と `ld` の起動クラッシュは解消した。

一方、`gcc.elf hello.c` はまだ完了していない。

- `gcc.elf` 自体は起動する
- `/bin/cc1.elf` までは起動する
- ただし現時点では `cc1` が早い段階で
  - `#PF(User): read/page-not-present at 0x0000000000000000`
  - `RIP: 0x0000000000000000`
  で落ちる
- そのため `gcc -> cc1 -> as -> ld -> hello` の通し復旧は未完了

次に優先する作業:

- `cc1` の null RIP を逆引きし、`setlocale` / locale / startup / callback 初期化周辺を切り分ける
- `gcc.elf hello.c` の通しを復旧し、Day12 の `hello world` self-host path を current tree で再成立させる

追加の追跡:

- `cc1` の `__global_locale+0xe8` が指す callback は file image 上では `0x00c52260 (__ascii_mbtowc)` で、バイナリ上の初期値自体は壊れていない
- `toplev_main` 早期の間接 call 候補として
  - `host_hooks`
  - `lang_hooks`
  があり、これらも file image 上では有効な関数ポインタを持っている
- このため、現時点の主な疑い先は
  - 実行時にどこかで `.data` / `.bss` が壊れている
  - あるいは null 間接 call の発生箇所が `locale` 以外の別 callback である
  のどちらかになった

追加で入れた暫定対策:

- `kernel/vmm.c` の user page fault dump に `RSP` 先頭 4 qword を追加
  - 次回の `cc1` 再現時に return address を拾い、どの call site から `RIP=0` へ落ちたか逆引きするため
- `user/gcc.c` の frontend 呼び出しは `/bin/cc1.elf` から `/bin/cc1` に変更
  - 旧 GCC frontend は `argv[0]` や実行名に依存する経路が残るため、歴史的な実名へ寄せた
- `Makefile` でも `rootfs/bin/cc1` と `rootfs/bin/cc1.elf` の両方を current build から同期するようにした
- `ports/orthos-gcc.sh` は古い `build/user/syscalls.o` を見ていたため、current の `user/syscalls.o` / `build/newlib/user/syscall_wrap.o` を参照するよう修正
- toolchain 再リンク時のみ `-Wl,--wrap=signal` を有効化し、`user/syscalls.c` の `__wrap_signal` から OrthOS 側 `sigaction` wrapper を使うようにした
  - `cc1` は起動直後に複数の `signal()` を呼ぶため、newlib 内部の reent 依存経路を一度外して差分を見る

## さらに追記: musl 版 toolchain の着手結果

優先順位を修正し、`newlib` toolchain の延命ではなく `musl` 版 `gcc/as/ld/cc1` の成立を先に進める方針へ戻した。

今回追加したもの:

- `user/gcc-musl.elf`
- `user/as-musl.elf`
- `user/ld-musl.elf`
- `user/cc1-musl.elf`
- `toolchain-musl` target

また、`user/syscalls_musl.c` に以下を追加した。

- `sigaction`
- `sigprocmask`
- `sigpending`
- `__wrap_signal`

これは `cc1` / `binutils` が起動直後に `signal()` を使うためで、musl 側でも OrthOS の signal syscall へ最低限つなぐ必要があった。

確認結果:

- `user/gcc-musl.elf` はビルド成功
- ただし `user/as-musl.elf` / `user/ld-musl.elf` / `user/cc1-musl.elf` は未成立

詰まり方の本質:

- `cc1` は GCC 本体だけでなく
  - `libiberty`
  - `libcpp`
  - `gmp`
  - `mpfr`
  - `mpc`
  まで `newlib` 前提でビルドされており、link 時に
  - `_impure_ptr`
  - `__errno`
  - `_ctype_`
  といった `newlib` 固有 ABI を要求する
- `as` / `ld` も同様に
  - `libbfd`
  - `libopcodes`
  - `libiberty`
  が `newlib` ABI を引きずっており、musl libc だけ差し替えても解決しない

つまり、現時点で分かったのは:

- `gcc/as/ld/cc1` の musl 化は「トップの実行ファイルを musl で再リンクするだけ」では足りない
- 少なくとも `binutils` / `gcc` 配下の補助ライブラリ群も musl 前提で再ビルドする必要がある
- `newlib` 互換 shim を無理に増やすより、依存ライブラリごと musl で組み直す方が筋が良い

次に優先する作業:

- `binutils` 側:
  - `libbfd`
  - `libopcodes`
  - `libiberty`
  を musl 前提で再ビルドできるように build tree を分離する
- その後で `as-musl` / `ld-musl` の起動確認
- `cc1-musl` はその次
